## 추가 수정: obsvdata UPSERT → DO NOTHING 변경

- obsvdata(증분 동기화)도 매 실행마다 `execution_id`를 덮어씀
- 이전 실행의 IF 레코드가 최신 execution_id로 갱신되어 이전 실행 상세 조회 불가

  - `useUpsert = config.isFullCopy() || isTimeRangeExecution` 으로 판단
  - PG: `ON CONFLICT DO NOTHING` / MySQL: `INSERT IGNORE`

| 조건 | PG | MySQL |
|------|-----|-------|
| fullCopy=true (제원) | `ON CONFLICT DO UPDATE SET ...` | `ON DUPLICATE KEY UPDATE ...` |
| fullCopy=false + 일반 실행 (관측데이터) | `ON CONFLICT DO NOTHING` | `INSERT IGNORE` |
| fullCopy=false + RESYNC (기간 지정) | `ON CONFLICT DO UPDATE SET ...` | `ON DUPLICATE KEY UPDATE ...` |

## LOADER 성능 최적화: JDBC batch UPSERT 전환

**배경**: LOADER 파이프라인(IF_RSV → SEC_* 타겟 테이블)이 10,704건 처리에 15분 소요.
JPA `merge()` + `IDENTITY` 제너레이터 = JDBC 배치 불가, RESYNC 시 레코드별 SELECT (N+1), 레코드별 IF 상태 UPDATE.

### 병목 분석

| 순위 | 병목 | 영향 |
|------|------|------|
| 1 | JPA merge() + IDENTITY = 배치 INSERT 불가 | 매 INSERT마다 개별 SQL |
| 2 | RESYNC N+1 SELECT (`findSecObsvdataIdByUniqueKey`) | ~10K 개별 SELECT |
| 3 | 레코드별 IF 상태 UPDATE (`markAsProcessed`) | ~10K 개별 UPDATE |
| 4 | 레코드별 Link 업데이트 (`updateLinkLastSync`) | ~400 개별 트랜잭션 |

### 수정 내용

**1. SecJewon.java** - `@UniqueConstraint(obsv_code)` 추가
- ON CONFLICT UPSERT에 필요한 UK 생성 (hbm2ddl=update로 자동 생성)

**2. TargetRepositoryService.java** - JDBC batch UPSERT 메서드 3개 추가
- `batchUpsertJewon()`: 1000건 단위 JDBC batch, `ON CONFLICT (obsv_code) DO UPDATE`
- `batchUpsertObsvdata()`: 1000건 단위 JDBC batch, `ON CONFLICT (obsv_code, obsv_date, obsv_time) DO UPDATE`
- `batchUpsertLinks()`: 전체 1회 JDBC batch, `ON CONFLICT (obsv_code) DO UPDATE`
- 기존 JPA 메서드는 조회 API용으로 유지

**3. DaejeonLoadStep.java** - 전면 리팩토링
- 제원: `deleteJewonByObsvCodes()` + JPA `saveAllJewon()` → JDBC `batchUpsertJewon()` 1회 (DELETE 제거)
- 관측데이터: per-record RESYNC SELECT + JPA merge → JDBC `batchUpsertObsvdata()` (ON CONFLICT가 자동 처리)
- IF 상태: per-record `markAsProcessed()` → 성공/실패 ID 수집 후 `batchMarkAsProcessed()` 1~2회
- Link: per-obsv_code JPA merge → `batchUpsertLinks()` JDBC batch 1회

**4. dmz-bojo-loader.yml** - `batch-size: 100 → 1000`
