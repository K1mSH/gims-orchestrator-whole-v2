## 1. 페이징 처리

페이징은 두 가지 방식이 있습니다. 데이터 규모에 따라 적절한 방식을 선택하세요.

### 방식 A: 메모리 페이징 (소규모 데이터) >> 부장님 언급 방식

전체 리스트를 조회한 뒤 `subList`로 잘라서 반환합니다.

```java
public Page<UnregUgwtrFcltEntity> findWithMemoryPaging(SearchVO vo, Pageable pageable) {
    // 1) 전체 데이터를 한 번에 조회
    List<UnregUgwtrFcltEntity> list = queryFactory
            .selectFrom(entity)
            .where(buildCondition(vo))
            .orderBy(entity.regDt.desc())
            .fetch();

    // 2) 메모리에서 페이지 잘라내기
    int start = (int) pageable.getOffset();
    int end = Math.min(start + pageable.getPageSize(), list.size());

    return new PageImpl<>(list.subList(start, end), pageable, list.size());
}
```

- **장점**: count 쿼리가 별도로 필요 없음, 구현이 간단
- **단점**: 전체 데이터를 매번 메모리에 올림 → **데이터가 수천 건 이상이면 성능 저하**
- **적합**: 코드 테이블, 소규모 마스터성 데이터 (수백 건 이하)

### 방식 B: DB 페이징 (대규모 데이터) — 권장

DB에서 offset/limit으로 해당 페이지만 조회하고, count 쿼리를 별도 실행합니다.

```java
public Page<UnregUgwtrFcltEntity> findWithDbPaging(SearchVO vo, Pageable pageable) {
    // 1) 해당 페이지 데이터만 조회
    List<UnregUgwtrFcltEntity> content = queryFactory
            .selectFrom(entity)
            .where(buildCondition(vo))
            .orderBy(entity.regDt.desc())
            .offset(pageable.getOffset())
            .limit(pageable.getPageSize())
            .fetch();

    // 2) 전체 건수 별도 조회
    long total = queryFactory
            .select(entity.count())
            .from(entity)
            .where(buildCondition(vo))
            .fetchOne();

    return new PageImpl<>(content, pageable, total);
}
```

- **장점**: 필요한 데이터만 DB에서 가져옴 → 메모리 효율적
- **단점**: count 쿼리가 추가로 실행됨 (조건이 복잡하면 느려질 수 있음)
- **적합**: 관측 데이터, 이력 테이블 등 **수천~수만 건 이상인 경우**

### 선택 기준

| 기준 | 메모리 페이징 | DB 페이징 |
|------|:---:|:---:|
| 예상 전체 건수 < 500 | O | - |
| 예상 전체 건수 >= 500 | - | O |
| 검색 조건이 자주 바뀜 | - | O |
| 정렬이 복잡함 | - | O |

## 2. Native Query 지양

### 왜 Native Query를 피해야 하는가?

| 항목 | JPQL / QueryDSL | Native Query |
|------|:---:|:---:|
| 컴파일 타임 오류 검출 | O | X |
| 엔티티 필드명 변경 시 자동 반영 | O | X |
| DB 방언(dialect) 독립 | O | X |
| 영속성 컨텍스트 관리 | O | 제한적 |
| 1차 캐시 활용 | O | X |
| Dirty Checking | O | X |

### 비교 예시

```java
// ① Native Query (지양)
@Query(value = "SELECT * FROM ob_obsvjewon_tb WHERE obsv_code = :code", nativeQuery = true)
ObsvJewonEntity findByObsvCodeNative(@Param("code") String code);
// 문제점:
// - obsv_code 컬럼명이 바뀌면 런타임에서야 에러 발견
// - 반환값이 영속성 컨텍스트에 제대로 관리되지 않을 수 있음
// - DB를 PostgreSQL → Oracle로 바꾸면 SQL 수정 필요할 수 있음


// ② JpaRepository 메서드 이름 규칙 (권장)
// 이 정도 단순 조건은 메서드 이름만으로 충분
Optional<ObsvJewonEntity> findByObsvCode(String obsvCode);
// 장점: 쿼리를 직접 작성할 필요 없음, 필드명 변경 시 컴파일 에러로 바로 잡힘


// ③ QueryDSL + BooleanBuilder (복잡한 동적 조건일 때 권장 → 5장 참고)
public ObsvJewonEntity findByCondition(String code, String type) {
    BooleanBuilder where = new BooleanBuilder();
    if (code != null) where.and(obsvJewon.obsvCode.eq(code));
    if (type != null) where.and(obsvJewon.obsvType.eq(type));

    return queryFactory
            .selectFrom(obsvJewon)
            .where(where)
            .fetchOne();
}
// 장점: 타입 안전 + 동적 쿼리에 강함
```

### Native Query가 허용되는 경우

- DB 고유 함수를 반드시 써야 할 때 (예: PostgreSQL `jsonb` 연산)
- 대량 벌크 INSERT/UPDATE로 성능이 중요한 경우 (우리 서버의 경우 5초 이내로 통신해야함)
- 기존 레거시 SQL을 급하게 이식해야 할 때 (임시 허용, 리팩토링 대상 명시)

## 3. BooleanBuilder — 동적 WHERE 조건

QueryDSL의 `BooleanBuilder`를 사용하면 검색 조건을 동적으로 조합할 수 있습니다.
위 ③ 예시의 확장판입니다.

### 기본 사용법

```java
public List<ObsvJewonEntity> search(ObsvSearchVO vo) {
    BooleanBuilder where = new BooleanBuilder();

    // null이 아닌 조건만 추가됨
    if (vo.getRegionCode() != null) {
        where.and(obsvJewon.regionCode.eq(vo.getRegionCode()));
    }
    if (vo.getObsvType() != null) {
        where.and(obsvJewon.obsvType.eq(vo.getObsvType()));
    }
    if (vo.getKeyword() != null) {
        where.and(
            obsvJewon.obsvName.contains(vo.getKeyword())
            .or(obsvJewon.obsvCode.contains(vo.getKeyword()))
        );
    }
    if (vo.getFromDate() != null && vo.getToDate() != null) {
        where.and(obsvJewon.regDt.between(vo.getFromDate(), vo.getToDate()));
    }

    return queryFactory
            .selectFrom(obsvJewon)
            .where(where)
            .orderBy(obsvJewon.regDt.desc())
            .fetch();
}
```

### 헬퍼 메서드로 정리

조건이 많아지면 별도 메서드로 분리하면 가독성이 좋아집니다.

```java
private BooleanBuilder buildCondition(ObsvSearchVO vo) {
    BooleanBuilder builder = new BooleanBuilder();

    addIfNotNull(builder, vo.getRegionCode(), obsvJewon.regionCode::eq);
    addIfNotNull(builder, vo.getObsvType(), obsvJewon.obsvType::eq);
    addIfNotBlank(builder, vo.getKeyword(), obsvJewon.obsvName::contains);

    if (vo.getFromDate() != null && vo.getToDate() != null) {
        builder.and(obsvJewon.regDt.between(vo.getFromDate(), vo.getToDate()));
    }

    return builder;
}

// 범용 헬퍼
private <T> void addIfNotNull(BooleanBuilder builder, T value,
                               Function<T, BooleanExpression> expr) {
    if (value != null) {
        builder.and(expr.apply(value));
    }
}

private void addIfNotBlank(BooleanBuilder builder, String value,
                            Function<String, BooleanExpression> expr) {
    if (value != null && !value.isBlank()) {
        builder.and(expr.apply(value));
    }
}
```

### 주의사항

- `BooleanBuilder`에 아무 조건도 추가하지 않으면 **WHERE 절 없이 전체 조회**됩니다 → 의도한 것인지 확인
- OR 조건은 `where.and(a.or(b))` 형태로 감싸야 합니다 (그냥 `.or()`를 쓰면 전체 조건이 OR로 연결될 수 있음)

## 4. JpaRepository 내장 메서드 활용

### 기본 원칙

Spring Data JPA는 **메서드 이름만으로 쿼리를 자동 생성**합니다. 단순한 조회는 직접 쿼리를 작성하지 말고, 내장 기능을 최대한 활용하세요.

### 단계별 활용

#### Step 1: JpaRepository 기본 메서드

```java
public interface ObsvJewonRepository extends JpaRepository<ObsvJewonEntity, String> {
    // 이미 제공되는 메서드들:
    // save(entity)            — INSERT or UPDATE
    // findById(id)            — PK 단건 조회
    // findAll()               — 전체 조회
    // findAll(Pageable)       — 페이징 조회
    // deleteById(id)          — PK 삭제
    // count()                 — 전체 건수
    // existsById(id)          — 존재 여부
}
```

#### Step 2: 메서드 이름 기반 쿼리 (Query Method)

```java
public interface ObsvJewonRepository extends JpaRepository<ObsvJewonEntity, String> {

    // WHERE obsv_code = ?
    Optional<ObsvJewonEntity> findByObsvCode(String obsvCode);

    // WHERE region_code = ? AND obsv_type = ?
    List<ObsvJewonEntity> findByRegionCodeAndObsvType(String regionCode, String obsvType);

    // WHERE obsv_name LIKE '%keyword%'
    List<ObsvJewonEntity> findByObsvNameContaining(String keyword);

    // WHERE reg_dt BETWEEN ? AND ? ORDER BY reg_dt DESC
    List<ObsvJewonEntity> findByRegDtBetweenOrderByRegDtDesc(LocalDateTime from, LocalDateTime to);

    // WHERE region_code = ? (페이징)
    Page<ObsvJewonEntity> findByRegionCode(String regionCode, Pageable pageable);

    // 건수 조회
    long countByObsvType(String obsvType);

    // 존재 여부
    boolean existsByObsvCode(String obsvCode);
}
```

#### Step 3: 복잡한 조건은 CustomRepository

메서드 이름이 너무 길어지거나, 동적 조건이 필요하면 **Custom Repository**로 분리합니다.

```java
// 1) Custom 인터페이스 정의
public interface ObsvJewonRepositoryCustom {
    Page<ObsvJewonEntity> search(ObsvSearchVO vo, Pageable pageable);
}

// 2) QueryDSL 구현
public class ObsvJewonRepositoryImpl implements ObsvJewonRepositoryCustom {

    private final JPAQueryFactory queryFactory;

    @Override
    public Page<ObsvJewonEntity> search(ObsvSearchVO vo, Pageable pageable) {
        BooleanBuilder where = new BooleanBuilder();

        if (vo.getRegionCode() != null) {
            where.and(obsvJewon.regionCode.eq(vo.getRegionCode()));
        }
        if (vo.getKeyword() != null) {
            where.and(obsvJewon.obsvName.contains(vo.getKeyword()));
        }

        List<ObsvJewonEntity> content = queryFactory
                .selectFrom(obsvJewon)
                .where(where)
                .offset(pageable.getOffset())
                .limit(pageable.getPageSize())
                .fetch();

        long total = queryFactory
                .select(obsvJewon.count())
                .from(obsvJewon)
                .where(where)
                .fetchOne();

        return new PageImpl<>(content, pageable, total);
    }
}

// 3) 기본 Repository에서 Custom 상속
public interface ObsvJewonRepository
        extends JpaRepository<ObsvJewonEntity, String>, ObsvJewonRepositoryCustom {
    // 기본 메서드 + Custom 메서드 모두 사용 가능
}
```

### 판단 흐름

```
단순 PK 조회?          → findById()
단순 조건 1~2개?       → findByXxx() 메서드 이름
조건이 3개 이상 / 동적? → CustomRepository + QueryDSL
```

## 5. Entity PK 생성 유형 — 복합키 vs 대리키

PK가 단일 컬럼이 아닌 경우, 두 가지 선택지가 있습니다.

### 방식 A: 별도 SN 컬럼 추가 (대리키) — 권장

```java
@Entity
@Table(name = "ob_obsvdata_tb",
       uniqueConstraints = @UniqueConstraint(columnNames = {"obsv_code", "obsv_date"}))
public class ObsvDataEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long sn;                    // 대리키 (auto increment)

    @Column(name = "obsv_code")
    private String obsvCode;            // 비즈니스 키 1

    @Column(name = "obsv_date")
    private LocalDate obsvDate;         // 비즈니스 키 2

    // ... 기타 필드
}
```

### 방식 B: 기존 복합 PK 사용 (`@EmbeddedId`)

```java
@Embeddable
@EqualsAndHashCode
public class ObsvDataId implements Serializable {
    @Column(name = "obsv_code")
    private String obsvCode;

    @Column(name = "obsv_date")
    private LocalDate obsvDate;
}

@Entity
@Table(name = "ob_obsvdata_tb")
public class ObsvDataEntity {

    @EmbeddedId
    private ObsvDataId id;              // 복합 PK

    // ... 기타 필드
}
```

### 우리 프로젝트 기준

- 대리키(SN) 방식을 기본으로 사용하되 `save()` 만으로는 INSERT/UPDATE 구분이 안 되므로, **비즈니스 키 기준 존재 여부 검증 로직을 별도 구현**해야 한다.

```java
// 기존 테이블 (복합 PK) — INSERT/UPDATE 시 비즈니스 키 검증 필요
public void saveObsvData(ObsvDataDto dto) {
    // 1) 비즈니스 키로 기존 데이터 존재 여부 확인
    Optional<ObsvDataEntity> existing = repository.findByObsvCodeAndObsvDate(
            dto.getObsvCode(), dto.getObsvDate());

    if (existing.isPresent()) {
        // 2-a) UPDATE: 기존 엔티티에 값 세팅
        ObsvDataEntity entity = existing.get();
        entity.updateFrom(dto);
        // dirty checking으로 자동 UPDATE
    } else {
        // 2-b) INSERT: 새 엔티티 생성
        ObsvDataEntity entity = ObsvDataEntity.from(dto);
        repository.save(entity);
    }
}

// 신규 테이블 (대리키 SN) — save() 만으로 처리 가능
public void saveFacility(FacilityDto dto) {
    FacilityEntity entity = FacilityEntity.from(dto);
    // sn == null → INSERT, sn != null → UPDATE
    repository.save(entity);
}
```
