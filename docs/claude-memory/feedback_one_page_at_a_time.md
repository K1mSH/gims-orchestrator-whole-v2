페이지 단위 디자인/UI 작업 (KRDS 마이그레이션 등) 은 한 페이지씩 완결하고, 사용자가 브라우저로 직접 화면 확인 + 디테일 수정 피드백을 받은 후에야 다음 페이지로 진입한다.

**Why:**
- 5/8 KRDS 작업 중 `agents/[id]` 헤더/탭은 OK 였으나 InfoTab 미정리로 부조화 발생 → 같은 사이클에서 마저 정리해야 했음
- 한 페이지 안에 KRDS 톤과 legacy 톤 혼재 시 사용자가 비교 검증하기 어려움
- 톤/디테일 (예: 사이드바 이모지 제거, stat 카드 사이드 라인 굵기, cron input small) 은 코드만 보고는 판단 불가, 화면으로 확인해야 결정 가능
- 한 번에 여러 페이지 묶어 진행하면 피드백 누적 시 어느 페이지에 적용해야 할지 추적 곤란

**How to apply:**
- 페이지 단위로 작업 분할 (예: Phase 1=login, Phase 2=api-provide 목록 등)
- 한 Phase 완료 → 타입체크 + 인라인 스타일 카운트 + 사용자에게 화면 확인 요청
- 사용자 디테일 수정 피드백 → 같은 Phase 내에서 처리
- 사용자 OK 신호 (예: "통과", "다음 가자") 후에만 다음 Phase 진입
- 한 Phase 안에 컴포넌트 + 페이지가 같이 묶여 있으면 함께 처리 (부조화 방지)
- **도메인 묶음 진행 우선** — "인라인 카운트 가벼운 순"보다 "같은 도메인 페이지 연달아" 가 사용자 화면 검증 흐름에 자연스러움. 예: api-provide 목록 → api-provide/new → api-provide/[id] → api-collect 목록 → api-collect/[id]. 도메인 내에서 자연스러운 사용자 흐름(목록→등록→상세) 순으로 진행
