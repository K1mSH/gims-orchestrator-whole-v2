계획서(dev_plan/*.md)를 작성하는 것과 "작업을 진행해도 된다"는 별개다. 계획서를 쓴 직후 바로 코드 수정 → 빌드 → 재기동으로 넘어가지 말 것.

- "계획서 쓰고 ~ 해볼까?" / "계획서 간단하게라도 쓰고 진행하자" 같은 말도 — 계획서를 먼저 보여주고, 사용자가 그걸 읽고 OK 한 뒤에 코드. 애매하면 "계획서 이거 — 진행할까요?" 한 줄 더 물을 것.
- 특히 auth / 운영 영향 / DB 상태 변경 / 여러 모듈 걸치는 작업은 더 보수적으로.
- 이미 메모리에 `feedback_no_guess_ask_first` / `feedback_no_scope_creep` 가 있고 작업 규칙에 "사용자가 확인 후 진행하라고 할 때까지 코드 수정 금지" 가 있음 — 이게 또 안 지켜져서 사용자가 반복적으로 지적함.

**Why:** 사용자가 계획 단계에서 방향을 틀거나 범위를 줄이고 싶을 수 있는데, 계획서 쓰자마자 구현해버리면 그 기회가 사라짐. 신뢰 문제.

**How to apply:** 코드 수정 전 = ① 계획서 작성 → ② 사용자에게 보여주고 명시적 승인 받기 → ③ 그 다음 코드. ②를 건너뛰지 말 것. 이미 진행 중인 게 있으면 솔직히 현 상태 보고하고 사용자 결정 기다리기.
