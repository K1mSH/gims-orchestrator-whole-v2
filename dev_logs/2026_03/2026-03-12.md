### 작업 내용

### 수정 파일 (이전 세션 포함)

#### sync-agent-common
- `step/ConditionBuilder.java` — 새 파일. 동적 WHERE 절 빌더 (merge + build 패턴)
- `step/ExecutionCondition.java` — 새 파일. 조건 VO (column, operator, value, value2)
- `step/ConditionOperator.java` — 새 파일. 연산자 enum (EQ, NEQ, GT, GTE, LT, LTE, BETWEEN, IN, LIKE, IS_NULL, IS_NOT_NULL)
- `step/ExecutionOptions.java` — conditions 필드 추가, hasConditions(), buildWhere() 메서드
- `step/SourceToIfStep.java` — fetchSimpleCopy()에 ConditionBuilder 통합, CUSTOM_STAGING 바이패스 로직, skipLinkUpdate 플래그 설정
- `pipeline/PipelineRunner.java` — conditions 파싱 로직 (rawParams → ExecutionCondition 리스트)

#### sync-agent-bojo (DMZ)
- `loader/step/DefaultLoadStep.java` — isResyncExecution 통합, buildMergedConditions(), conditions-aware 쿼리
- `loader/repository/TargetRepositoryService.java` — findIfRsvJewonForResync(), findIfRsvObsvdataForResync() (Native SQL + ConditionBuilder)
- `config/DynamicEntityManagerService.java` — getTargetDbType() 추가

#### sync-agent-bojo-int (Internal)
- `loader/step/InternalLoadStep.java` — isResyncExecution 통합, buildMergedConditions(), JDBC ConditionBuilder 적용, Link 테이블 스킵 로직

#### sync-orchestrator/backend
- `execution/ExecutionDto.java` — TriggerRequest에 conditions 필드 추가
- `execution/ExecutionController.java` — conditions 전달 및 빈 conditions 400 거부 (안전가드)
- `execution/ExecutionService.java` — triggerExecutionInternal에 conditions 파라미터 추가, Agent 요청에 conditions 포함

#### sync-orchestrator/frontend
- `types/index.ts` — ExecutionCondition 인터페이스, CONDITION_OPERATORS 상수
- `lib/api.ts` — trigger 함수에 conditions 파라미터 추가
- `app/agents/[id]/page.tsx` — WHERE 조건 입력 UI (컬럼/연산자/값, 추가/삭제)

### E2E 테스트 결과

| 테스트 | 결과 | 비고 |
|--------|------|------|
| conditions 전달 (Orchestrator → Agent) | PASS | conditions가 Agent request에 정상 포함 |
| NONEXISTENT obsv_code EQ | PASS | 0건 반환 (필터링 동작 확인) |
| obsv_code LIKE 'GPM-305%' | PASS | 0건 (daejeon에 해당 패턴 없음) |
| 빈 conditions 거부 (400) | PASS | Orchestrator 안전가드 동작 |
| 일반 증분 실행 (조건 없음) | PASS | 402건, CUSTOM_STAGING 경로 정상 |
| CUSTOM_STAGING 바이패스 | PASS | 로그: "bypassing CUSTOM_STAGING, using SIMPLE_COPY" |
| skipLinkUpdate | PASS | 로그: "skipLinkUpdate=true — Link 테이블 갱신 스킵" |

### 핵심 설계 포인트
- **defaults → merge(conditions) → build(WhereClause)**: 디폴트 조건(link_status 등)과 사용자 조건을 merge
- **CUSTOM_STAGING 바이패스**: conditions 실행 시 Link 기반 증분(LinkTableObsvDataFetcher) 불필요 → SIMPLE_COPY 강제
- **skipLinkUpdate**: 재동기화 시 과거 데이터로 link 테이블 오염 방지
- **3중 안전가드**: 프론트 disable + Orchestrator 400 + Agent 예외
