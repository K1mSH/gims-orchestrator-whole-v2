## 작업 내용

### 1. 파이프라인 실행 시 DB 정보 흐름 검토 및 수정

기존 암호문 통신 전환(3/17)의 테스트 과정에서 connection-info 흐름의 구조적 문제 3건 발견 → 수정 완료.

**문제 1: Agent → Orchestrator 직접 fallback (Proxy 우회)**
- Agent의 `resolveFromProxy()`가 Proxy 실패 시 Orchestrator에 직접 connection-info 요청
- Proxy가 자격증명 전달의 유일한 경로여야 하는데 우회 가능 → 보안 문제
- **수정**: Orchestrator fallback 코드 제거, `proxy-url` 미설정 시 즉시 예외
- `fetchFromOrchestrator()` 메서드 삭제, `getJdbcTemplate()`의 Orchestrator 조회도 제거

**문제 2: URL 경로 불일치**
- Proxy (common): `GET /api/datasource/connection-info/{id}`
- Orchestrator: `GET /api/datasources/{id}/connection-info`
- Agent가 하나의 URL 패턴으로 양쪽에 요청하니 둘 중 하나만 맞는 상황
- **수정**: Orchestrator 경로를 표준으로 통일

**문제 3: Proxy가 복호화된 평문을 Agent에게 서빙**
- common 모듈의 connection-info 엔드포인트가 `ProxyDataSourceService.getJdbcTemplate()` → 복호화 → 평문 반환
- **수정**: common의 connection-info 엔드포인트 삭제 + Proxy에 패스스루 전용 Controller 신규 추가
- `ConnectionInfoController`: Orchestrator 응답을 복호화 없이 그대로 Agent에게 전달
