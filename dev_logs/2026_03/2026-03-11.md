## retentionDays 음수 방어 코드 추가

### 작업 목적
- retentionDays에 음수/0 입력 시 `LocalDate.now().minusDays(음수)` → 미래 날짜 계산 → 현재 데이터 전체 삭제되는 버그 방어
- 4개 계층(프론트 ~ Agent 실행)에 다중 방어 적용

### 주요 변경사항

#### 1. 프론트엔드 — 입력 단계 차단
- `<input type="number" min={1}>` + `Math.max(1, ...)` 적용
- 음수/0 입력 시 자동으로 1로 보정
- **파일**: `sync-orchestrator/frontend/components/agent/InfoTab.tsx`

#### 2. Orchestrator API — 저장 전 검증
- `updateRetentionConfig()` 내 JSON 파싱 후 targets 순회
- retentionDays < 1이면 `IllegalArgumentException` throw
- **파일**: `sync-orchestrator/backend/.../domain/agent/AgentService.java`

#### 3. Agent Controller — 파싱 시 검증
- `parseFromBody()` 내 retentionDays 파싱 후 1 미만이면 예외
- 500 응답으로 Orchestrator에 알림
- **파일**: `sync-agent-common/.../controller/DataRetentionController.java`

#### 4. Agent Service — 실행 직전 최종 방어
- `executeCleanup()` 실행 시 retentionDays < 1이면 해당 테이블 skip
- 경고 로그 출력 + 결과에 에러 메시지 포함
- **파일**: `sync-agent-common/.../service/DataRetentionService.java`

## Loader 실행모드별 Step 교체 구조

### 작업 목적
- `executionModeId`에 따라 다른 Step 구현체로 파이프라인 실행하는 구조 구축
- 향후 새로운 실행 모드(수동 지역별 Load 등) 추가 시 Step 구현체만 추가하면 되는 확장성 확보
- DaejeonLoadStep → DefaultLoadStep 리네이밍 (보조망 전체 Loader이므로)

### 설계 결정
- **Step은 monolithic 유지** — jewon/obsvdata 분리하지 않음 (향후 경계 변경 대비)
- **모드별 Step 구현체 교체** — PipelineConfig에서 모드별 Runner 미리 등록
- **공통 처리 로직 헬퍼 추출** — 변환/UPSERT/IF상태/SyncLog 재사용
- **PipelineRegistry 복합키** — `(agentCode, modeId)`, 미등록 모드는 default fallback

### 주요 변경사항

#### 1. PipelineRegistry — 복합키 `(agentCode, modeId)` 지원
- `Map<String, PipelineRunner>` → `Map<String, Map<String, PipelineRunner>>`
- `getRunner(agentCode, modeId)` 추가, modeId 없으면 default fallback
- 기존 시그니처 하위호환 오버로드 유지 (RCV/SND 수정 불필요)
- **파일**: `sync-agent-bojo/.../config/PipelineRegistry.java`, `sync-agent-bojo-int/.../config/PipelineRegistry.java`

#### 2. PipelineService — modeId 추출 + Runner 선택
- params에서 `executionModeId` 추출 → `getRunner(agentCode, modeId)` 호출
- **파일**: `sync-agent-bojo/.../pipeline/PipelineService.java`, `sync-agent-bojo-int/.../pipeline/PipelineService.java`

#### 3. LoaderStepHelper — 공통 처리 로직 추출 (DMZ bojo)
- `processJewon()` — 변환 + batch UPSERT + IF 상태 업데이트
- `processObsvdata()` — 변환 + batch UPSERT + IF 상태 업데이트
- `saveSyncLog()` — SyncLog 매핑 단위 저장
- `applyObsvCodeFilter()` — obsv_code 필터 적용
- **파일**: `sync-agent-bojo/.../loader/step/LoaderStepHelper.java` (신규)

#### 4. DaejeonLoadStep → DefaultLoadStep 리네이밍 + 리팩터링
- 클래스명 변경, 공통 로직을 LoaderStepHelper 호출로 교체
- stepId: `daejeon-load` → `default-load`, stepName: `보조망 데이터 적재 (default)`
- **파일**: `sync-agent-bojo/.../loader/step/DefaultLoadStep.java` (리네이밍), `DaejeonLoadStep.java` (삭제)

#### 5. LoaderPipelineConfig — 모드별 등록 구조
- `register(agentCode, "LOADER", "default", runner)` 명시적 모드 등록
- **파일**: `sync-agent-bojo/.../config/LoaderPipelineConfig.java`

#### 6. application.yml — stepId/stepName 업데이트
- **파일**: `sync-agent-bojo/src/main/resources/application.yml`

### 수정된 파일 목록

## Agent Credentials → Proxy 경유 (보안 아키텍처 변경)

### 작업 목적
- **문제**: Orchestrator → Agent 실행 요청 시 DB credentials(host, port, username, password)를 HTTP 평문으로 전송
- **해결**: Agent가 직접 Proxy에서 DB 접속정보를 해석하도록 변경 (Proxy = DB credential 전담)
- **추가**: HikariCP 풀 하드닝, 실행 전 풀 상태검사, YML 전체 Jasypt 암호화, Proxy API Key 인증

### 계획 문서
- `dev_plan/2026_03/11/agent-credentials-via-proxy.md`

**테스트 기준**:

**테스트 기준**:
- [ ] Proxy에 API Key 없이 요청: `curl http://localhost:8083/api/datasource/connection-info/test` → 401
