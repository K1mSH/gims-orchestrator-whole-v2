### 핵심 원리
- execution_id는 항상 target에만 존재 (source에는 없음)
- source 조회 흐름: `target WHERE execution_id = ? → source_refs 수집 → source 조회`
- 분기 로직 (3단계):
  1. source에 `source_refs` 컬럼이 없으면 → PK 파싱 매칭 (외부 DB — RCV)
  2. source에 `source_refs` 있고, target의 source_refs 값과 동일 → source_refs IN 매칭 (Loader)
  3. source에 `source_refs` 있지만, target의 source_refs 값이 다름 → PK 파싱 매칭 (SND)
- 판별: target source_refs 샘플 1건으로 source에서 매칭 존재 여부 확인

### 추가 수정 (E2E 테스트 중 발견)

#### SND source 추적 0건 → 수정
- **문제**: SND의 target(if_snd) source_refs = `["D:dsId:tbId:{sec_jewon.id}"]` (PK 기반 새 생성)
  source(sec_jewon) source_refs = `["E:dsId:tbId:{외부PK}"]` (값이 다름)
  → hasColumn 분기만으로는 SND에서 source_refs IN 매칭 실패 (0건)
- **수정**: `buildSourceFilter()`에 샘플 1건 매칭 확인 추가
  - target source_refs 값을 source에서 찾아보고 없으면 PK 파싱으로 fallback
- **패턴 정리**:
  - Loader: target이 IF의 source_refs를 그대로 복사 → source_refs IN 매칭
  - SND/RCV: target이 source의 PK 기반 새 source_refs 생성 → PK 파싱 매칭
