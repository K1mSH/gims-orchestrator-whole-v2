# 2026-03-19 작업 일지

## 작업 내용

### 1. Proxy/Agent API 보안 강화

**문제**: Proxy가 common 모듈을 스캔하면서 cleanup(DELETE) 등 불필요한 엔드포인트가 노출됨. Agent에는 API Key 인증이 없어 무방비 상태.

**수정 A: common 컨트롤러 `@ConditionalOnProperty` 적용**
- 4개 컨트롤러에 활성화 조건 추가 (기본값 false)
- Agent: 전부 활성 / Proxy: execution-data, datasource만 활성
- 새 컨트롤러 추가 시 명시적으로 켜지 않으면 비활성 → 구조적 방어

**수정 B: ApiKeyFilter를 common으로 통합**
- Proxy 전용이던 ApiKeyFilter를 common에 신규 생성 (ConditionalOnProperty)
- Proxy 자체 ApiKeyFilter 2개 삭제 (DMZ/Internal)
- Agent(bojo/bojo-int)에도 api-key 인증 적용
- `/health`만 인증 제외, `/api/**` 전부 X-API-Key 필수

**수정 C: `/debug/datasources` 엔드포인트 제거**
- 4개 모듈 HealthController에서 디버그 엔드포인트 삭제
- 호스트/포트/DB명 노출 방지, 로그로 충분히 대체 가능

### 2. InternalBojoLoadStepFactory 테이블명 기본값 수정

- `targetLinkTable` 기본값: `link_ngwis` → `tm_gd980002` (YAML 정의와 일치)
- `targetResultTable` 기본값: `tm_gd980002` → `tm_gd970101` (YAML 정의와 일치)
- 기존 버그: link/result 테이블명이 뒤바뀌어 Internal Loader 실행 시 SQL 에러

### 3. 조건실행 테이블 바인딩 (conditions tableName)

**문제**: conditions가 테이블 구분 없이 모든 Step에 동일하게 적용돼, jewon에 없는 `obsv_date` 컬럼 조건이 걸리면 SQL 에러.

**수정:**
- `ExecutionCondition`에 `tableName` 필드 추가
- `PipelineRunner.buildExecutionOptions()`에서 `tableName` 파싱 추가 (근본 원인)
- `SourceToIfStep.fetchSimpleCopy()`에서 tableName 필터링 → 자기 source-table 조건만 적용
- `DmzBojoLoadStep`, `InternalBojoLoadStep`의 `buildMergedConditions()`에도 동일 필터링
- 프론트 `types/index.ts`, `page.tsx`에서 `condTableSelections`를 conditions에 포함
- `tableName == null`이면 모든 Step에 적용 (하위호환)

### 4. ConditionBuilder 날짜 타입 캐스팅

**문제**: PostgreSQL date 컬럼에 BETWEEN/GTE/LTE 등으로 String 값 바인딩 시 타입 불일치 에러.

**수정:** `castIfDate()` 헬퍼 — `yyyy-MM-dd` 패턴이면 `java.sql.Date`로 변환하여 바인딩.
