## 작업 내용

### 1. api-collector LOOKUP 파생 컬럼 기능

**목적**: 외부 API 응답에 없는 컬럼을 내부 공통코드 기반으로 파생 생성
- 예: `orignl_url` → 도메인 추출 → GIMS 공통코드(NGW_0118) → 언론사명

**설계**:
- 기존 `ApiFieldMapping` 엔티티에 LOOKUP 필드 통합 (별도 엔티티 없이)
- `isDerived` 플래그로 1:1 매핑과 파생 컬럼 구분
- 정규식(`extractPattern` + `extractGroup`)으로 키 추출 — 프리셋 제공 (도메인/숫자/괄호)
- LOOKUP API URL은 `application.yml`의 `lookup.common-code-url` 설정, UI에서는 그룹코드만 입력
- `lookupMatchType` 필드 예비 (향후 CONTAINS 등 확장)

**구현**:
- `ApiFieldMapping` 엔티티: LOOKUP 전용 필드 9개 추가
- `TransformType` 확장: SUBSTRING, TRIM, REPLACE, DEFAULT_VALUE, LOOKUP
- `DataTransformer`: 단순 변환 4종 구현
- `LookupService` 신규: 내부 API 호출 → Map 캐싱 → 정규식 키 추출 → LOOKUP
- `ApiExecutionService`: 일반/파생 매핑 분리 처리, LOOKUP Map 사전 로딩
- `CreateRequest`에 적재 설정 필드 추가 (등록 시 한번에 저장)

### 2. API 키 바인딩 (GIMS 본체 연동)

**목적**: API 키를 GIMS 본체에서 조회하여 사용, 키 관리는 본체에서만

**설계**:
- 기존 `ApiParam` 활용 — `paramType=HEADER`로 헤더 통합
- `staticValue`에 본체 API 키 ID만 저장 (실제 키 값 X)
- 실행 시점에 본체 API 호출 → ID로 실제 키 조회 → 헤더/파라미터에 주입
- `description`에 `🔑 서비스명` 저장하여 API키 참조 식별

**구현**:
- `ApiCallService`: 🔑 파라미터 감지 → `resolveApiKey(id)` → 본체 API 조회 + 캐싱
- `MockApiController`: API 키 목록 Mock API 추가
- `ApiEndpointController`: API 키 목록 조회 프록시 엔드포인트
- `application.yml`: `lookup.api-key-url` 설정 추가

### 3. 프론트엔드 헤더/파라미터 통합

**변경**: 기존 헤더 JSON 입력 → `ApiParam(paramType=HEADER)`로 통합
- UI에서 헤더/파라미터를 `paramType`으로 섹션 분리 표시
- 데이터는 같은 `params` 배열에 저장
- 기존 `endpoint.headers` JSON 읽기 로직 제거

**API 키 선택 UI**:
- 값 유형: `직접입력 / 🔑 API키 / 동적` 3분기
- API키 선택 → 본체 API 키 목록 드롭다운 (serviceName + 만료일)
- 선택 시 ID만 저장, 설명에 `🔑 서비스명` 표시
- 등록 폼(page.tsx) + 상세 수정(InfoTab.tsx) 양쪽 적용
