## 4. Link UPSERT 최적화 (COALESCE + batch)

### 배경: 현행(레거시)과의 비교

현행 시스템(Oracle/iBatis)은 **관측소 중심 루프**로 동작:
```
FOR EACH 관측소:                    ← N회 반복
    SELECT getLinkData              ← N회 DB 호출 (기존 Link 조회)
    INSERT/UPDATE link 분기          ← N회 DB 호출
    SELECT getObsvdate              ← N회 DB 호출 (관측데이터 조회)
    INSERT ALL (batch 1000건)        ← ceil(건수/1000) x N회
```
→ 총 **최소 3N + alpha회** DB 호출

우리 구현은 **데이터 중심 플랫 구조**로 개선:
- Phase 1: 사전 벌크 로드 (2회) → 메모리 캐시로 N+1 제거
- Phase 2: 전체 PENDING 1회 SELECT → 통합 batch INSERT
- **Phase 3(Link)만 N+1 잔존** → 이번에 개선

### 개선 1: COALESCE로 getLink SELECT 제거

**문제**: 기존 frst_date/frst_time 보존을 위해 매번 SELECT 후 Java에서 분기
```java
// AS-IS: 관측소별 SELECT (N회 DB 호출)
Map<String, Object> existingLink = targetRepo.getLink(targetLinkTable, obsvCode);
String frstDate = existingLink != null ? existingLink.get("frst_date") : dateStr;
```

**해결**: SQL의 `COALESCE(기존값, 신규값)`으로 판단을 DB에 위임
```sql
-- TO-BE: SELECT 없이 SQL이 자동 판단
ON CONFLICT (obsrvt_id) DO UPDATE SET
  frst_date = COALESCE(tm_gd980002.frst_date, EXCLUDED.frst_date),
  frst_time = COALESCE(tm_gd980002.frst_time, EXCLUDED.frst_time)
```
- 기존 Link 존재(UPDATE 실행 시): `COALESCE(기존값, 신규값)` → 기존값 유지
- 신규 INSERT 시: frst_date/frst_time에 VALUES 값이 그대로 들어감
- **N회 SELECT → 0회**

### 개선 2: 개별 UPSERT → batch UPSERT

**문제**: 관측소별 개별 `targetRepo.upsertLink()` 호출 (N회 DB 호출)
```java
// AS-IS: 관측소별 개별 UPSERT (4,006회 DB 호출)
for (Map.Entry<String, String[]> entry : maxDateTimePerCode.entrySet()) {
    targetRepo.upsertLink(targetLinkTable, ...);  // 1회씩
}
```

**해결**: 파라미터 리스트를 모아서 obsvdata batch INSERT와 동일한 패턴으로 처리
```java
// TO-BE: 리스트 모아서 1000건씩 batch (5회 DB 호출)
List<Object[]> linkRows = new ArrayList<>();
for (...) {
    linkRows.add(new Object[]{obsvCode, spotId, dateStr, timeStr, now, dateStr, timeStr});
}
targetRepo.batchUpsertLink(targetLinkTable, linkRows);
```
- **N회 UPSERT → ceil(N/1000)회**

### 효과 요약

| 구간 | 변경 전 | 변경 후 | 감소율 |
|------|---------|---------|--------|
| getLink SELECT | 4,006회 | 0회 | 100% |
| upsertLink | 4,006회 | 5회 | 99.9% |
| Phase 3 합계 | **8,012회** | **5회** | 99.9% |

### 수정 파일
- `sync-agent-bojo-int/.../loader/repository/GimsTargetRepository.java`
  - `upsertLink()` (개별) → `batchUpsertLink()` (batch + COALESCE) 교체
  - `getLink()` 삭제 (COALESCE로 불필요)
- `sync-agent-bojo-int/.../loader/step/InternalLoadStep.java`
  - Phase 3: for문 내 개별 호출 → 리스트 모아서 `batchUpsertLink()` 1회 호출

## 5. Execution Mode (실행 방식) 개념 분리 및 구현

### 배경
- 기존 `agent_step_definition`은 파이프라인 내부 실행 단위 (jewon-extract 등)
- 실제로 필요한 건 "이 Agent를 어떤 방식으로 실행할 것인가"라는 상위 개념
- 예: 증분 적재 vs 전체 재적재 vs 특정 관측소만 — 모드 ID만 전달, 파라미터 없음

### 설계
- **새 테이블**: `agent_execution_mode` (agent_id, mode_id, mode_name, is_default 등)
- **Agent YAML**: `execution-modes` 섹션에 모드 정의
- **Agent API**: `GET /api/pipeline/{agentCode}/execution-modes` 엔드포인트
- **실행 흐름**: Orchestrator → Agent 실행 요청에 `executionModeId` 필드 추가
- **Agent 내부**: `ExecutionOptions.modeId`로 각 Step에서 분기 처리 가능
