### 회귀 — sync-agent-common AutoConfiguration 통합

부팅 시도 시 backend / api-provider / api-collector 모두 동일 에러:
```
Parameter 0 of constructor in ...SecurityConfig required a bean of type
'com.sync.agent.common.config.JwtCookieAuthFilter' that could not be found.
```

원인 = JwtCookieAuthFilter / JwksClient 가 `com.sync.agent.common.*` 패키지인데 검증자 모듈 메인 클래스(`com.sync.orchestrator` / `com.gims.provider` / `com.infolink.collector`) component scan 범위 밖.

#### 해법 후보
| 옵션 | 처리 | 영향 |
|---|---|---|
| A | 3개 모듈 메인 클래스에 `@ComponentScan(basePackages={"자기", "com.sync.agent.common"})` | 메인 수정 영향 큼 |
| B | 3개 모듈에 직접 `@Bean` 정의 | 코드 중복 3개소 |
| **C ✅** | **sync-agent-common 자체에 `@AutoConfiguration` + `META-INF/spring/AutoConfiguration.imports`** | **단일 책임. 검증자 모듈 yml 토글만 박으면 자동 동작** |

사용자 메모리 `feedback_dependency_consistency.md` 정수에 정합 (모듈끼리 의존성 통일, common 단일 책임).

#### 산출물
| 파일 | 내용 |
|---|---|
| `sync-agent-common/.../auth/AuthVerifierAutoConfiguration.java` | `@AutoConfiguration` + `@ConditionalOnProperty(auth.jwks-url)` JwksClient + `@ConditionalOnProperty(jwt.cookie.enabled=true)` JwtCookieAuthFilter. `@ConditionalOnMissingBean` 으로 모듈 자체 등록 회피 |
| `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` | `com.sync.agent.common.auth.AuthVerifierAutoConfiguration` (Spring Boot 2.7+ 새 표준) |

### 사용자 발견 fix — 첫 로그인 후 헤더 stale (curl 검증으로 안 잡힘)

**증상**: admin/alice 어떤 사용자든 첫 로그인 직후 AppHeader 가 사용자 이름 노출 안 함 — "로그인" 버튼 그대로. 강제 새로고침(Ctrl+F5) 시점부터 정상.

**원인**: `useCurrentUser` 가 모듈 단위 캐시. `/login` 페이지 마운트 시 `useEffect` 가 `/auth/me` 호출 → 401 → `cached=null, fetched=true` 박힘. 로그인 성공 후 `router.replace('/')` 만 호출 → AppHeader 마운트 시 `fetched=true` 라 재 fetch 안 함 → cached=null 그대로.

같은 증상이 발생하는 모든 흐름:
- 미인증 첫 진입 → middleware /login redirect → 로그인 (사용자가 본 시나리오)
- 로그아웃 후 다른/같은 사용자로 재로그인
- 본인 비번 변경 또는 탈퇴 후 재로그인

**fix** (`app/login/page.tsx`):
```tsx
const { mutate } = useCurrentUser();
...
await authApi.login(username, password);
await mutate();        // ← cached 비우고 fetch 강제
router.replace(next);
router.refresh();
```

**검증** — 다른 stale 가능 지점 점검 결과 모두 정합 (logout 의 `clear()`, 본인 탈퇴, refresh 자동 갱신, 서버 재기동 후 cookie 살아있는 상태 진입). login 한 곳만 fix 필요.

**교훈** — curl E2E 37개로는 잡을 수 없는 클라이언트 사이드 캐시 stale. UI 검증 필요 케이스 명시.

### 2. Frontend Route Group 분리

#### 발견 — 사용자 보고 stale layout
- 시나리오: 사용자가 로그인된 상태 + Backend/Auth 재기동 → axios 401 → interceptor `window.location.href = /login?next=...` (full reload) → 사용자 admin 다시 입력 + 로그인 → **sidebar + header 둘 다 사라진 화면** (F5 시 정상 복원)
- 원인 = `AppShell.tsx` 의 path 분기:
  ```tsx
  const pathname = usePathname();
  if (pathname === '/login') return <>{children}</>;   // ← layout 없는 분기
  return <Sidebar/> + <AppHeader/> ...
  ```
- `router.replace('/')` 직후 client navigation transition 동안 `usePathname()` 이 잠깐 stale `/login` 반환 → AppShell 이 plain children 만 렌더 → sidebar + header 둘 다 사라짐. F5 시 SSR 새 path 로 정상 복원 = 패턴 일치

#### 해법 — Next.js Route Group (정공법)

기존:
```
app/
  layout.tsx          → root: <AppShell>{children}</AppShell> (모든 페이지 감쌈, path 분기로 갈라짐)
  page.tsx, login/, agents/, ...
```

변경:
```
app/
  layout.tsx          → root: <html><body>{children}</body></html> (단순)
  login/page.tsx      → /login (root layout 만)
  (main)/             → route group, URL 영향 0 (괄호 명명)
    layout.tsx        → <AppShell>{children}</AppShell> (이 그룹 모두 자동 묶음)
    page.tsx          → /
    agents/, datasources/, api-collect/, api-provide/, executions/, users/
```

→ AppShell 의 path 분기 자체 제거. `/login` 은 root layout 만 통과 (sidebar/header 자동 없음). 그 외 모두 (main)/layout 자동 적용 (timing 무관).
