### 원인 (`SourceToTargetStep.fetchSimpleCopy()`)
- 보조 RCV = jewon step + obsvdata step. conditions 실행이면 둘 다 SIMPLE_COPY 경로.
- `sigungu` 는 `sec_jewon_view` 에만 있는 컬럼. tableName=`sec_jewon_view` 조건은 jewon step 에만 적용(80건), obsvdata step 은 적용 조건 0개.
- 옛날 코드: 적용 조건 0개 + `!fullCopy` → 디폴트 `link_status IN ('PENDING','RESYNC','FAILED')` 폴백. `sec_obsvdata_view` 는 평소 CUSTOM_STAGING(link_ngwis 조인)으로 도는 뷰라 `link_status` 컬럼 없음 → 크래시.
- (tableName 미지정 시엔 obsvdata step 이 `WHERE sigungu=...` 시도 → `column "sigungu" does not exist` — 같은 계열의 다른 실패)
- 참고: DMZ Loader(`DmzBojoLoadStep`)는 이미 "조건 있지만 이 테이블 대상 조건 없으면 skip" 처리됨. RCV(`SourceToTargetStep`)만 미커버였음.

### 수정 (`infolink-agent-common/.../step/SourceToTargetStep.java` `fetchSimpleCopy()`)
1. `execConditions` 산출 시 tableName 필터에 더해 **컬럼 존재 필터** 추가: condition.column 이 이 source-table 의 조회 컬럼(`columns`)에 없으면 제외 (jewon 전용 컬럼이 obsvdata step 으로 새지 않게)
2. **execution-level conditions 실행인데 step-level 적용 조건 0개면 → `return new ArrayList<>()`** (빈 리스트 → readCount=0/writeCount=0/SUCCESS 경로). 디폴트 `link_status` 폴백 도달 안 함. (DMZ Loader 와 동일 시맨틱)
3. legacy startTime/endTime 경로는 불변. 회귀 없음 (조건 없는 일반 실행·공통 컬럼 조건 동작 그대로).

## 오후 — retention 스케줄 테스트 (셋업 → 검증 → 원복)

사용자가 retention(고위험 기능) 을 육안 검증하고 싶어함 → 테스트 베드 셋업 + 스케줄 경로 검증.

- **DMZ `sec_obsvdata`**: synthetic 50행 (`obsv_code LIKE 'RTN-%'`, 5 버킷 × 10: 2024-06-15 / 2025-12-15 / 2026-03-15 / 2026-04-10 / 2026-05-07). agent 17 retention_config = `{sec_obsvdata/obsv_date/365}`.
- **Oracle `pm_gd970201`**: synthetic 50행 (`execution_id='RTN-SETUP'`, 동일 5 버킷). `OBSRVN_DT` DATE 타입 확인. agent 20 retention_config = `{PM_GD970201/OBSRVN_DT/365}`.
- 백엔드를 `RETENTION_CRON='0 */2 * * * *'` (2분마다) 로 임시 재기동 → `DataRetentionScheduler` 발동 관찰.
- 결과: 14:28 → agent 17 **10건 삭제** (B1/2024, cutoff=2025-05-12, `<`) → 5041→5031, 실데이터(2026) 보존. 14:38 → agent 20 **10건 삭제** (B1/2024) → 74678→74668. 실제 행 사라짐 확인(`obsv_date='2024-06-15'` → 0건, `RTN-B2-%` 멀쩡). 스케줄 경로(스케줄러 → DB config 읽기 → agent `/api/cleanup/{code}` 호출) 정상. ("대상 Agent 1개" 였던 건 agent 17 의 config 가 중간에 UI 에서 `targets:[]` 로 비워져서 — 스케줄러는 targets 비면 skip)
- **retention 에러 케이스 정리** (코드 확인): `DataRetentionService.executeCleanup` = `DELETE FROM {table} WHERE {dateColumn} < ?` (`?=Timestamp`), 테이블별 try/catch (한 테이블 에러나도 전체 안 죽고 error 메시지+0건). 에러 안 나는 조건 = ① table/column 실존 ② **dateColumn 이 date/timestamp 타입** (varchar 날짜는 PG 에서 `operator does not exist` 에러, Oracle 은 NLS 운빨) ③ retentionDays ≥ 1 (0/음수 = skip+에러메시지) ④ yml retention-candidates 등록(VER-016) ⑤ DB 연결. `<` 라 cutoff 당일 데이터는 보존.
- **string 날짜 컬럼 지원** = 미지원, 별도 enhancement 후보. yml candidate 에 `dateColumnType` (예: `STRING_YYYYMMDD`) 같은 거 도입하면 cutoff 를 컬럼 포맷 문자열로 만들어 사전식 비교(고정폭 ISO 정렬순 포맷) — 추후 필요 시.
- **원복 완료**: agent 17·20 retention_config → `{enabled:false, targets:[]}`. synthetic 행 삭제 (DMZ `sec_obsvdata` 4991 / Oracle `pm_gd970201` 74628, RTN 0). 백엔드 평소대로 재기동 → `retention.cron` `0 0 2 * * *` 원복.
