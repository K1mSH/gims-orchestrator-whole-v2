## api-collect sourceFieldType (backend + frontend)

### 배경
매핑 화면 재진입 시 "타입" 컬럼이 빈 값 — `getFieldType(testResult?.responseTree, ...)` 가 testResult 없으면 빈 값. `endpoint.fieldMappings` 에 타입을 저장하지 않았기 때문.

### 변경
- **Backend** (`infolink-api-collector`):
  - `ApiFieldMapping` entity — `source_field_type VARCHAR(20)` 컬럼 추가
  - `ApiEndpointDto.FieldMappingRequest/Response` — `sourceFieldType` 추가
  - `ApiEndpointService.saveMappings` — sourceFieldType 저장 + **빈 값/누락 시 기존 DB 값 보존** (items 빈 응답 케이스 대비)
- **Frontend**:
  - `types/api-collect.ts` — `ApiFieldMapping`/`ApiFieldMappingRequest` 에 sourceFieldType
  - `MappingTab.tsx` — MappingRow 에 sourceFieldType, 로드 시 사용, 호출 시 응답에서 채움, 저장 시 전송, 표시 fallback (응답 → 저장값 → '-')
  - `'null'`/`'unknown'` 타입은 의미없는 값으로 빈 처리 (DB 에 "null" 저장 안 함)
- DDL: dev 환경 Hibernate auto-ddl 로 컬럼 자동 추가됨. **운영 배포 시 별도 DDL 필요** (`ALTER TABLE api_field_mapping ADD COLUMN source_field_type VARCHAR(20)`)

### 한계
- items 배열이 빈 응답이면 응답 트리에 leaf 타입 없음 → 그 호출에선 타입 추출 못함. 데이터 있는 시점에 호출 한 번 + 저장 필요 (외부 API 운빨). 수동 입력 옵션은 미도입 (사용자 결정).
