### FAILED 레코드 재처리 추가

**변경:** `SourceToIfStep.buildPendingSelectSql()`에서 조회 조건에 `'FAILED'` 추가
- 기존: `WHERE link_status IS NULL OR link_status IN ('PENDING', 'RESYNC')`
- 변경: `WHERE link_status IS NULL OR link_status IN ('PENDING', 'RESYNC', 'FAILED')`
- 이유: FAILED 건도 재처리 시도해야 관리자가 로그로 인지 가능. 무시하면 영구 누락됨

## 서버 사이드 Agent Health Check 스케줄러

### 배경
- 대시보드/에이전트 관리에서 상태 확인이 주기적으로 필요
- 프론트에서 13개 Agent 개별 health check를 계속 던지는 건 비효율적
- 현재 대시보드는 DB 상태만 읽고 있으나, 자동 갱신이 없어 Agent가 죽어도 감지 불가

### 해결: Orchestrator 백엔드 스케줄러
- 서버에서 30초마다 전체 Agent health check → DB 반영
- 프론트는 기존대로 DB SELECT 한 방 (변경 없음)
- 브라우저 탭 수와 무관하게 Agent당 30초 1회만 호출

### 수정 파일
**sync-orchestrator/backend:**
- `agent/AgentHealthScheduler.java` - 신규 (서버 사이드 스케줄러)
  - `@Scheduled(fixedDelay=30000, initialDelay=10000)`
  - `CompletableFuture`로 전체 Agent 병렬 health check
  - RUNNING 상태 제외 (실행 흐름에서 관리)
  - 상태 변경 시에만 DB write + 로그
  - 기존 수동 상태확인 버튼 유지 (즉시 확인용)

| orchestrator backend 빌드 | 성공 |
| 첫 health check (기동 10초 후) | online=13, offline=0 정상 |
