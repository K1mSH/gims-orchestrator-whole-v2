# 2026-02-20 작업 일지

## 작업 내용: conflict-key 설정 옵션 구현 + SND 파이프라인 수정

외부 DB에 obsv_code 중복이 존재하는 제원(jewon) 테이블의 UPSERT 충돌 기준을
obsv_code에서 source_refs로 변경. YAML 설정 기반으로 선택적 적용.

---

### 1. conflict-key 구현 (common + bojo)

**배경**: 외부 DB 10개에서 같은 obsv_code를 가진 제원 레코드가 중복 존재.
기존 시스템에서 UPDATE 대신 INSERT로 데이터를 관리하는 업체가 있어
obsv_code를 UK로 사용할 수 없음. source_refs(`["E:dsId:tbId:pk"]`)는 출처별로
자연 유일하므로 UPSERT conflict 기준으로 적합.

**설계 원칙:**
- conflict-key는 선택적 설정 (YAML에 없으면 기존 primary-key 사용)
- 제원(jewon)만 적용, 관측데이터(obsvdata)는 기존 복합 PK 유지
- 기본 전략은 INSERT (DO NOTHING), fullCopy=true 또는 기간 지정 실행일 때만 UPSERT

**수정 파일 (common):**

| 파일 | 변경 내용 |
|------|-----------|
| `ExtractStepConfig.java` | `conflictKey` 필드 추가 (선택적) |
| `SourceToIfStep.java` | `buildUpsertSql()`: conflictKey 우선 사용, 로그에 conflict 정보 표시 |

**수정 파일 (bojo):**

| 파일 | 변경 내용 |
|------|-----------|
| `AgentDefinition.java` | `TableConfig`에 `conflictKey` 필드 추가 |
| `AgentConfigLoader.java` | YAML `conflict-key` 파싱 추가 (jewon, obsvdata 양쪽) |
| `RcvPipelineConfig.java` | jewonConfig builder에 `.conflictKey()` 전달 |
| `IfRsvSecJewon.java` | UK: `obsv_code` → `source_refs` |
| `SecJewon.java` | UK: `obsv_code` → `source_refs` (Loader target) |
| `TargetRepositoryService.java` | `batchUpsertJewon()`: `ON CONFLICT (source_refs)`, obsv_code UPDATE SET 제외 |
| RCV YAML 10개 | jewon 섹션에 `conflict-key: source_refs` 추가 |

### 2. SND 파이프라인 수정

**문제**: SND jewon의 `primary-key: obsv_code`인데 `if_snd_sec_jewon`에
obsv_code UK가 없어서 `ON CONFLICT (obsv_code)` 실패 (800건 전부 skip).

**원인**: SND의 source인 sec_jewon은 이미 외부 데이터가 합쳐진 상태이고,
SND는 단순 복사이므로 sec_jewon의 PK(id)를 그대로 사용하면 됨.

**수정:**

| 파일 | 변경 내용 |
|------|-----------|
| `dmz-bojo-snd.yml` | jewon `primary-key: obsv_code` → `id` |
