### 1. conflict-key 구현 (common + bojo)

**배경**: 외부 DB 10개에서 같은 obsv_code를 가진 제원 레코드가 중복 존재.
기존 시스템에서 UPDATE 대신 INSERT로 데이터를 관리하는 업체가 있어
obsv_code를 UK로 사용할 수 없음. source_refs(`["E:dsId:tbId:pk"]`)는 출처별로
자연 유일하므로 UPSERT conflict 기준으로 적합.

**설계 원칙:**
- conflict-key는 선택적 설정 (YAML에 없으면 기존 primary-key 사용)
- 제원(jewon)만 적용, 관측데이터(obsvdata)는 기존 복합 PK 유지
- 기본 전략은 INSERT (DO NOTHING), fullCopy=true 또는 기간 지정 실행일 때만 UPSERT

**수정 파일 (common):**

| 파일 | 변경 내용 |
|------|-----------|
| `ExtractStepConfig.java` | `conflictKey` 필드 추가 (선택적) |
| `SourceToIfStep.java` | `buildUpsertSql()`: conflictKey 우선 사용, 로그에 conflict 정보 표시 |

**수정 파일 (bojo):**

| 파일 | 변경 내용 |
|------|-----------|
| `AgentDefinition.java` | `TableConfig`에 `conflictKey` 필드 추가 |
| `AgentConfigLoader.java` | YAML `conflict-key` 파싱 추가 (jewon, obsvdata 양쪽) |
| `RcvPipelineConfig.java` | jewonConfig builder에 `.conflictKey()` 전달 |
| `IfRsvSecJewon.java` | UK: `obsv_code` → `source_refs` |
| `SecJewon.java` | UK: `obsv_code` → `source_refs` (Loader target) |
| `TargetRepositoryService.java` | `batchUpsertJewon()`: `ON CONFLICT (source_refs)`, obsv_code UPDATE SET 제외 |
| RCV YAML 10개 | jewon 섹션에 `conflict-key: source_refs` 추가 |

### 6. SourceToIfStep 버그 수정 3건

**버그 1: writeCount 부정확**
- `batchUpdate()` 반환값 `int[]` 무시하고 `writeCount += currentBatchSize`로 일괄 가산
- 실제 INSERT 실패해도 writeCount에는 성공으로 잡히는 문제
- 수정: `int[]` 각 원소를 검증하여 `SUCCESS_NO_INFO(-2)` 이상만 카운트

**버그 2: 복합 PK source status 업데이트 불가**
- `updateSourceLinkStatus()`가 단일 컬럼 `WHERE pk IN (?)` 방식만 지원
- obsvdata의 복합 PK(`obsv_code,obsv_date,obsv_time`)는 SQL 문법 오류 → catch에서 무시됨
- 수정: `WHERE (col1, col2, col3) IN ((?,?,?), ...)` ROW값 비교 지원

**버그 3: PK 타입 불일치로 source status 업데이트 0건**
- `buildRecordKey()`가 모든 PK값을 String으로 변환
- `WHERE id IN ('6841')` → bigint 컬럼에 String 바인딩 → PostgreSQL 암시적 캐스팅 불가, 매칭 0건
- `WHERE obsv_date IN ('2026-02-20')` → date 컬럼에 String → 동일 문제
- 수정: `id::text`(PG) / `CAST(id AS CHAR)`(MySQL)로 컬럼을 text 캐스팅하여 비교

**수정 파일:**

| 파일 | 변경 내용 |
|------|-----------|
| `sync-agent-common/.../step/SourceToIfStep.java` | writeCount 검증, 복합 PK 지원, text 캐스팅, dbType 파라미터 추가 |

### 12. Loader sync_log table_type 수정 (IF → SOURCE)

**문제**: Loader가 if_rsv 테이블을 `IF` 타입으로 기록 → 프론트에서 SOURCE 표시 없음, TARGET_IF로 잘못 표시되어 execution_id 불일치로 데이터 0건

**원인**: table_type을 물리적 위치(테이블 이름이 if_*니까 IF)로 적음. 파이프라인 논리적 역할(읽기=SOURCE)과 혼동.

**재발 방지 원칙**: `sync_log.table_type`은 항상 파이프라인 논리적 역할로 기록
- 읽기 대상 → SOURCE, 쓰기 대상 → TARGET
- IF는 SourceToIfStep 전용 (IF zone에 쓰는 RCV/SND만 사용)

| 파일 | 변경 내용 |
|------|-----------|
| `DaejeonLoadStep.java` | sync_log table_type `"IF"` → `"SOURCE"` (정상경로 + 에러경로, 총 2곳) |

**DB 보정**: 기존 Loader sync_log `UPDATE sync_log SET table_type = 'SOURCE' WHERE step_id = 'daejeon-load' AND table_type = 'IF'` (2건)

### 14. Loader TARGET → SOURCE 역추적 (진행 중)

**문제**: Loader TARGET(sec_jewon) 행 클릭 시 역추적 실패
- sec_jewon의 source_refs는 외부 DB PK(obsv_code) 포함: `["E:1:4:DJ-DJC-G1-0001"]`
- sourceTable이 sec_jewon으로 그대로 전달 → `WHERE id = 'DJ-DJC-G1-0001'` (id는 bigint, 타입 불일치)
- Loader의 실제 SOURCE는 if_rsv 테이블이므로, sourceTable 해석 + source_refs LIKE 검색 필요

**현재 수정 상태**: 코드 수정 완료, 빌드 완료, Agent 재시작 완료. **API 테스트 미완료**.
