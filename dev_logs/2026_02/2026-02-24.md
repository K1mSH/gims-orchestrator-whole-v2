## 완료 작업: 시간 범위 지정 실행(RESYNC) obsvdata UPSERT 실패 수정

### 이슈5: obsvdata 시간 범위 실행 시 `ON CONFLICT (id)` 사용 → unique constraint 위반
- **증상**: 시간 범위 지정(RESYNC) 실행 시 obsvdata `success=0, skip=6` + SyncLog에 에러
  ```
  INSERT INTO if_rsv_sec_obsvdata (...) ON CONFLICT (id) DO UPDATE SET ...
  ERROR: duplicate key value violates unique constraint "uk_if_rsv_sec_obsvdata_source_refs"
  ```
- **원인**: `RcvPipelineConfig.java`에서 obsvdata ExtractStepConfig 빌드 시 `.conflictKey()` 누락
  - jewon에는 `.conflictKey(jewonCfg.getConflictKey())` 있지만, obsvdata 양쪽 경로(CUSTOM_STAGING, SIMPLE_COPY) 모두 빠져있음
  - conflictKey가 null이면 `buildUpsertSql()`이 primaryKey(id)로 fallback → `ON CONFLICT (id)`
  - 새 auto-increment ID는 기존 행과 충돌하지 않아 INSERT 시도 → source_refs UK 위반
- **수정**: `RcvPipelineConfig.java` obsvdata 빌드 양쪽에 `.conflictKey(obsvCfg.getConflictKey())` 추가
- **결과**:
  - 대전: obsvdata `성공=6, 실패=0` (gwdep=99.99 정상 UPDATE)
  - 인포로컬: obsvdata `성공=6, 실패=0` (gwdep=88.88 정상 UPDATE)
  - IF 테이블에서 12건 모두 `remark='update-test'` 값 정상 반영 확인

### Loader + SND 전 구간 검증
- Loader: jewon 1200, obsvdata 12건 모두 SUCCESS → Target 테이블 정상 적재
- SND: jewon 4000, obsvdata 12건 모두 SUCCESS → IF_SND 테이블 정상 적재
- 데이터 전파 확인: 대전 gwdep=99.99, 인포로컬 gwdep=88.88이 IF_RSV → Target → IF_SND까지 전파
- Trace 확인: RCV/Loader/SND 모든 구간에서 순방향(trace), 역방향(trace-source) 정상 동작

## 완료 작업: 선택적 Step 실행 인프라 구축

### 목적
실행옵션에서 사용자가 특정 Step만 선택하여 실행할 수 있는 인프라 구축.
Agent가 보유한 Step 메타데이터를 Orchestrator에 등록하고, 실행 시 선택된 Step만 실행하는 흐름.

### 전체 흐름
```
Agent 기동 → Step 메타데이터 PipelineRegistry에 등록
  ↓
Orchestrator → GET /api/pipeline/{agentCode}/step-definitions → DB 저장
  ↓
실행옵션 UI → 시간범위 + Step 체크박스 표시
  ↓
실행 트리거 → selectedStepIds: ["jewon-extract"] 전달
  ↓
PipelineRunner → selectedStepIds에 포함된 Step만 실행, 나머지 SKIPPED
```

### FAILED 케이스 분석
- 내부망 RCV(SIMPLE_COPY)에서는 **정상 운영 시 FAILED 거의 발생하지 않음**
- FAILED 발생 조건: target DB 접속 불가, 테이블 미존재, UK 충돌(conflictKey 설정 오류), 데이터 타입 불일치
- 정상 운영 상태에서는 **SKIP(변경 없음) 또는 SUCCESS** 둘 중 하나

### 발견된 이슈
- target 테이블 미존재 시: UPSERT 전체 실패 → `skip=전체건수`로 처리, source `link_status=FAILED`로 갱신되지만 **Step 자체는 SUCCESS로 보고됨** (에러가 batch에서 catch되어 skip 처리)
- Orchestrator step callback 404: `Failed to notify step finish: 404 /api/callback/step` → Orchestrator에 해당 콜백 API 미구현 (기능 영향 없음, 로그 경고만)
