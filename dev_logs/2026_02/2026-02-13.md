## v2 Orchestrator 엔티티/DB 스키마 정합성 수정

### 목적
v2 Orchestrator가 v1이 생성한 기존 DB 스키마와 호환되지 않아 Hibernate DDL 에러로 기동 실패하는 문제 해결.
v1의 엔티티 구조를 기준으로 v2 전체 백엔드를 정렬.

### 핵심 변경사항

#### Agent 엔티티 PK 변경
- **AS-IS (v2)**: `String agentId` 단일 PK
- **TO-BE (v1 정합)**: `Long id` (auto-increment PK) + `String agentCode` (unique 비즈니스 키) + `String datasourceTag`

#### ExecutionHistory 컬럼 수정
- **AS-IS**: `agent_id` (String) 컬럼
- **TO-BE**: `agent_code` (String) 컬럼 (v1 DB 스키마 일치)

#### AgentType 열거형 수정
- **AS-IS**: `RELAY`, `LOADER_STANDARD`, `LOADER_CUSTOM`
- **TO-BE**: `RCV`, `SND`, `LOADER` (v1 DB 데이터 호환)

#### Step/Pipeline 죽은 코드 제거
- `CallbackController`: `/step` 엔드포인트 삭제
- `CallbackService`: `handleStep()` 메서드 삭제
- `CallbackDto`: `StepCallbackRequest`, `StepResultDto` 삭제
- `ExecutionController`: step 관련 엔드포인트 삭제

### 수정 파일 목록

**Agent 도메인:**
- `Agent.java` - PK 구조 변경 (Long id + String agentCode + datasourceTag)
- `AgentRepository.java` - `JpaRepository<Agent, Long>`, findByAgentCode 추가
- `AgentDto.java` - Long id/agentCode 분리, agentCode 자동생성
- `AgentService.java` - Long id 기반, generateAgentCode() 추가
- `AgentController.java` - @PathVariable Long id
- `AgentType.java` - RCV, SND, LOADER 열거형

**Execution 도메인:**
- `ExecutionHistory.java` - agent_id → agent_code 컬럼
- `ExecutionHistoryRepository.java` - findByAgentCode* 메서드
- `ExecutionDto.java` - Long agentId + String agentCode 구조
- `ExecutionService.java` - Long id 기반, 죽은 코드 제거
- `ExecutionController.java` - Long id, 삭제된 엔드포인트 제거

**Callback 도메인:**
- `CallbackService.java` - agentCode 기반 조회
- `CallbackController.java` - /step 엔드포인트 삭제
- `CallbackDto.java` - Step 관련 DTO 삭제

**Schedule 도메인:**
- `ScheduleRepository.java` - findByAgentId(Long)
- `ScheduleService.java` - Long agentId
- `ScheduleExecutor.java` - getId() + getAgentCode()
- `ScheduleDto.java` - Long agentId + agentCode

**Chain 도메인:**
- `AgentChainDto.java` - Long agentId + agentCode
- (AgentChainService는 변경 불필요 - findById(Long) 자동 호환)

## agentCode 기반 Agent 등록 및 실행 라우팅 정합성 수정

### 목적
Agent 등록 시 Orchestrator가 `agentCode`를 자동생성(`{zone}-{tag}-{type}[-{seq}]`)하는데,
실제 Agent 프로세스 내부의 파이프라인은 YML에 정의된 `agent-code`(예: `dmz-bojo-rcv-01`)로 등록됨.
자동생성 코드와 YML 코드 불일치 시 PipelineRegistry에서 Runner를 못 찾아 런타임 에러 발생.
**해결**: Agent `/health` 조회 → 사용 가능한 agentCode 목록에서 선택하는 discover 방식으로 변경.

### 핵심 변경사항

#### Phase 1: Agent 쪽 `agentId` → `agentCode` 네이밍 통일
- **YML 12개**: `agent-id:` → `agent-code:` (dmz-bojo-rcv-01~10, loader, snd)
- **AgentDefinition.java**: `agentId` 필드 → `agentCode`
- **AgentConfigLoader.java**: `data.get("agent-id")` → `data.get("agent-code")`
- **PipelineRegistry.java**: 전체 리네이밍 (`getRegisteredAgentIds()` → `getRegisteredAgentCodes()`, `getAgentIdsByType()` → `getAgentCodesByType()`)
- **PipelineConfig 3개** (Rcv/Snd/Loader): `def.getAgentId()` → `def.getAgentCode()`
- **PipelineService.java**: `params.get("agentId")` → `params.get("agentCode")`, 변수/로그 전체
- **PipelineController.java**: `request.get("agentId")` → `request.get("agentCode")`, 응답 키
- **HealthController.java**: `getAgentIdsByType()` → `getAgentCodesByType()`

#### Phase 2: Orchestrator 백엔드
- **AgentDto.CreateRequest**: `@NotBlank agentCode` 필드 추가 (필수)
- **AgentService**: `generateAgentCode()`, `extractSeq()` 삭제. `request.getAgentCode()` 직접 사용 + 중복 체크. 신규 `discoverAgents(endpointUrl)` 메서드 추가
- **AgentController**: `GET /api/agents/discover?endpointUrl=...` 엔드포인트 추가
- **ExecutionService**: 실행 트리거 시 `request.put("agentCode", agentCode)` 추가

#### Phase 3: Orchestrator 프론트엔드
- **types/index.ts**: `AgentCreateRequest`에 `agentCode` 추가, `DiscoverAgent`/`DiscoverResponse` 타입 추가
- **lib/api.ts**: `agentApi.discover()` 메서드 추가
- **agents/page.tsx**: Agent 등록 폼을 discover 기반으로 전면 변경
  - endpointUrl 입력 → "에이전트 조회" → agentCode 목록 표시 → 선택 → 나머지 필드 입력

## 외부 DB 환경 실제 맞춤 및 테스트 데이터 구축

### 목적
외부 업체 DB를 실제 운영 환경과 동일하게 PostgreSQL/MySQL 이중 구성으로 맞추고,
10개 업체 DB 전체에 테스트 데이터(관측소 제원 + 관측 데이터)를 투입하여
Agent 동시 실행 테스트를 위한 기반 환경을 완성.

### 핵심 변경사항

#### 1. 외부 DB 이중화: PostgreSQL + MySQL
- **PostgreSQL (port 29000)**: 대전, 바이텍, 충남, 큰산 (4개)
- **MySQL Docker (port 29010)**: 인포월드_local/seoul, 하이드로넷_ara/idc/kyungnam/wonju (6개)
- MySQL 컨테이너: `gims_orchestrator_outer_mysql` (Docker Compose)
- 각 DB에 `k1m` 유저 (password: 1111) 생성, 전체 권한 부여

#### 2. 테이블 대소문자 정리
- **큰산만 대문자**: `SEC_JEWON_VIEW`, `SEC_OBSVDATA_VIEW` (대문자 컬럼명)
- **나머지 9개 소문자**: `sec_jewon_view`, `sec_obsvdata_view` (소문자 컬럼명)
- PostgreSQL 3개 DB + MySQL 6개 DB 테이블 재생성 완료

#### 3. link_ngwis 외부 DB에서 제거
- `link_ngwis` 테이블은 Agent IF DB(29001/dev)에만 존재해야 함
- 외부 업체 DB에서 전부 삭제

#### 4. Agent YML 업체 기반 네이밍 + 대소문자 분기

**파일명 변경:**
- **AS-IS**: `dmz-bojo-rcv-01.yml` ~ `dmz-bojo-rcv-10.yml` (번호 기반)
- **TO-BE**: `dmz-bojo-rcv-daejeon.yml`, `dmz-bojo-rcv-bytek.yml` 등 (업체명 기반)
- RCV 10개 + LOADER 1개 + SND 1개 = 12개 YML

**YML 구성:**
```yaml
# RCV 공통 구조 (daejeon 예시 - 소문자)
agent-code: dmz-bojo-rcv-daejeon
type: RCV
jewon:
  source-table: sec_jewon_view       # 소문자
  target-table: if_rsv_sec_jewon
  primary-key: obsv_code             # 소문자
  full-copy: true
  skip-source-status-update: true
obsvdata:
  source-table: sec_obsvdata_view    # 소문자
  target-table: if_rsv_sec_obsvdata
  primary-key: obsv_code,obsv_date,obsv_time
  date-column: obsv_date
  time-column: obsv_time
link:
  use-link-table: true
  table-name: link_ngwis
```
```yaml
# keunsan 예시 - 대문자
agent-code: dmz-bojo-rcv-keunsan
type: RCV
jewon:
  source-table: SEC_JEWON_VIEW       # 대문자
  primary-key: OBSV_CODE             # 대문자
  ...
obsvdata:
  source-table: SEC_OBSVDATA_VIEW    # 대문자
  primary-key: OBSV_CODE,OBSV_DATE,OBSV_TIME
  ...
link:
  table-name: LINK_NGWIS             # 대문자
```

| 업체 | DB타입 | source-table/column | link_ngwis |

| 하이드로넷_ara/idc/kyungnam/wonju | MySQL | 소문자 | 소문자 |
