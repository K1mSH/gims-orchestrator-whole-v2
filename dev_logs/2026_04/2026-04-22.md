### 오전 — 헤더 라우팅 + JdbcTemplate 전환

- **이슈 진단** (`dev_plan/2026_04/22/manage-datasource-header-routing.md`)
  - provide Agent가 PG 29006에 execution/sync_log 저장 → Proxy Internal(Oracle 29004)에서 못 읽음
  - "망당 한 DB" 전제가 provide 추가로 깨짐
- **설계 논의**
  - 여러 방향 검토 (Oracle 기본 datasource 전환 / 엔티티 이관 / 헤더 라우팅 등)
  - 최종: **"Agent = target 쪽" 규약 활용 → targetDatasourceId를 헤더로 주입**
  - 근거: 모든 파이프라인 Agent에서 target DB = 자기 JPA 기본 datasource = 관리 DB 성립
- **구현**
  - `ExecutionDataReader` 유틸 신규 (Execution/SyncLog JdbcTemplate 기반 조회)
  - `ExecutionDataController`: 13개 엔드포인트 `@RequestHeader("X-Manage-Datasource-Id")` 수신, JPA 호출 20지점 → JdbcTemplate 쿼리로 치환
  - `ExecutionService` (Orchestrator): `buildHeaders(Agent)` 공통 헬퍼, 8개 API 호출 `restTemplate.exchange`로 교체
  - Agent 쓰기(recordExecutionStart/Finish)는 JPA 유지 (자기 DB 쓰기라 라우팅 불필요)

### 오전 말 ~ 오후 — provide 파이프라인 디테일 이슈들

- `provide-tm-gd000203` 재실행 → 테이블별 처리현황 정상 표시 (헤더 라우팅 검증 성공)
- **연속 이슈 발견**:
  1. `/source` PK fallback `id` 에러 → ProvideLoadStep에 `source_pk_column=merge-key` 저장 추가
  2. `/target-if` `link_status` 컬럼 없음 에러 → 분석 결과 내부망 target 전반의 잠재 버그 (기존 bojo-int도 해당)
  3. `/trace-source` 역방향 매칭 실패 → 단방향 contains 가정이 provide 네이밍 방향과 반대

### 오후 — 방향 재정립 + 통합 개선

- **사용자 지적**: "수정 반복 중, 전체 Agent 케이스 포괄 설계 필요"
- **원칙 명문화**: `feedback_no_regression_organic.md` 메모리 신규 — "기존 기능 영향 없어야" 대전제
- **통합 계획서 작성** (`trace-and-lifecycle-coverage.md`)
  - 각 Agent의 테이블 구성/link_status 유무 전수 조사
  - 엔드포인트별 동작 매트릭스
  - "IF" 개념이 실제로는 **이름 규약**일 뿐 시스템 타입 아님 확인 (AgentTable.TableType에 IF 없음)
- **최종 설계 확정** (사용자 승인)
  - `/target-if` → `/target` 통합 (역할 중복 제거)
  - `/trace-source` 분기 3 내부만 개선 (prefix 분기 1,2는 기존 기능이라 유지)
  - 프론트 TARGET_IF 레거시 정리
- **구현**
  - 백엔드: `/target-if` 삭제 + `/trace-source` 분기 3에 target_tables 1순위 매칭 추가
  - 프론트: `getTargetIfData` 래퍼 삭제, 호출처 `getTargetData`로 교체, TARGET_IF 분기/라벨/변수 전체 제거
  - common JAR 재빌드 + 6개 모듈 배포, Proxy Internal 재기동

### 검증

- **provide**: 정방향(Source 행 클릭 → Target 추적) / 역방향(Target 행 클릭 → Source 추적) 모두 정상
- **bojo-int api-collect-loader**: 실행 성공 (IF_RSV_TM_GD014000 22204건, IF_RSV_TM_GD014001 381건 적재)
  - `NoClassDefFoundError: ConditionOperator` 초기 발생 → bojo-int JVM이 구 common JAR 로드 중 → 재기동 후 해결
- **bojo-loader 회귀 검증**: IF_RSV_SEC_OBSVDATA 데이터 없어 보류 (전체 파이프 정리 시점에 함께)

### 오후 추가 — Type A 추가 이식 착수

- **api-provider 중복 엔티티 4개 삭제** (entity/provide/) — 이관 아닌 "삭제"로 정리 완료
- **표준화 매핑 xlsx 변환** — NGW_테이블_원본 + 표준화사용테이블상세 시트를 JSON/TSV 5종으로 추출
  - `docs/Standardizedtable/_converted/` 아래 (NGW 운영 DB 1,248개, 표준화 매핑 367개 확인)
- **전략 문서** (`provide-source-table-strategy.md`): Type A/B 소스 테이블 × 운영DB 존재 × 표준화 매핑 매트릭스 + 이식 케이스 A/B/C 분류 + 이식 이름 원칙(표준화 후 11자리 + 환경부표준 컬럼)
- **SDE_NGWS / DBLINKUSR 스키마 생성** — 실운영 스키마 구조 재현 원칙, `scripts/ddl/internal-oracle/provide-source/00_create_schemas.sql`
- **TM_GD130001 DDL (A6 영향조사보고서)** — 멱등성 패턴 (BEGIN/EXCEPTION ORA-00955 무시, ALTER ORA-01430 무시, NULL→PENDING UPDATE)
- **16개 provide 엔티티 @Comment 일괄 추가** — 표준화 new_comment(한글) 기반
  - 테이블 레벨: `@org.hibernate.annotations.Table(appliesTo=, comment=)`
  - 컬럼 레벨: `@Comment("...")`
  - ddl-auto update가 기존 테이블 comment 변경 안 반영 → **16개 테이블 drop 후 재생성**으로 주석 완전 반영
- **TM_GD130001 E2E 실행 검증 완료** — 6건 PENDING 데이터 처리, 테이블별 처리현황/정방향/역방향 추적 정상
