### 오후 — trace-source 추가 버그 픽스 (VARCHAR PK leading-zero)

130001 trace-source가 `SOURCE_NOT_FOUND` 반환되는 이슈:

- 원인: `ExecutionDataController.typedValue()` 가 `"0000000001"` 같은 **VARCHAR2 PK 값을 `Long.parseLong`으로 변환** → Long 1 로 leading-zero 손실
- Oracle 쿼리 `WHERE ISVR_NO = 1` → 문자열 PK와 불일치 → 0건
- 단일 PK 분기(`:1151`)에 복합 PK 분기처럼 `isNumericPkColumn` 체크 추가
- 이후 FOUND + `pk=['0000000001']` 정상 복구
- 검증: 숫자 PK (112002 SN=22, 120001 GWEL_NO=10001), VARCHAR PK zero-padded (130001 `0000000001`), VARCHAR PK 순수 문자열 (130001 `Y-001`) 전 케이스 통과

### 오후 — auto-increment 제외 로직 개선 (방안 2)

다른 세션에서 `dev_plan/2026_04/23/auto-increment-pk-hardcoding-fix.md` 로 제기된 이슈:
- `AUTO_INCREMENT_PK_COLUMNS = ["id", "sn"]` 이름 블랙리스트 하드코딩
- 소스에 id/sn이라는 비즈니스 컬럼이 있을 경우 조용히 누락 위험

해결:
- `ExtractStepConfig.excludeInsertColumns` 필드 (null → 기본 `["id","sn"]`, YAML override 가능)
- `SourceToTargetStep.resolveExcludeInsertColumns()` 신규: **후보 ∩ 타겟 실제 컬럼** 교집합만 제외
- `fetchColumnsFromMetadata`를 타겟에도 재사용 (컬럼 이름만 조회 — Oracle/Tibero/PG 모두 안정)
- 하드코딩 상수 제거
- Factory 2곳에 `exclude-insert-columns` YAML 파싱 추가
- **IDENTITY 판별(방안 A)은 Tibero 드라이버 리스크로 오늘 제외**, 방안 B 기반 방안 2 채택

### 오후 말 — 로그 강화

약점(이름 기반 매칭의 한계)을 로그로 가시화:

- **INFO**: `exclude-insert-columns: candidates=[id,sn] (source=default), target matches=[sn] → final exclude=[sn]` — 결정 과정 투명화
- **WARN**: 제외된 컬럼에 **소스 첫 레코드의 값이 null 아니면** 1회 경고
  - 예: `Excluded column 'sn' has non-null source value (sample=21). If this is business data (not auto-increment), set exclude-insert-columns: [] in YAML to include it.`
  - auto-increment 가 아닌데 실수로 제외 당하는 상황 조기 감지

## 주요 논의 / 결정

- **source_refs 의미 재확인**: "source_refs는 소스 PK 값, 그뿐" — 내 초기 "대리키 vs 자연키" 설명이 혼동 유발. 사용자가 본질 정정
- **provide는 항상 UPSERT + UK 계층 전략** 유지 (메모리 `feedback_provide_layer_upsert`) — 대신 **RCV의 조건/기본 분기 로직은 자동 계승**
- **factory-key "source-to-if" 명칭 유지** (20개 YAML 호환). 이름 전환은 별도 이슈
- **IDENTITY 정밀 감지(방안 A)는 Tibero 드라이버 호환성 미검증으로 보류** — 방안 2 + 로그 강화로 대체
- **복합 PK 엣지 케이스** (타겟 id가 비즈니스 컬럼) 는 커버 안 함 — "비즈니스 컬럼을 id/sn으로 짓는 게 문제"이므로 로그 WARN 으로만 감지
- **레거시 API endpoint 단위로 타겟 테이블 분리**: 소스/컬럼이 동일해도 API endpoint가 다르면 타겟 분리 (A4가 112002와 거의 동일 구조임에도 별도 테이블 분리 결정) → 메모리 `feedback_provide_target_per_api`
- **A7 통일 패턴 예외 인정**: 소스 PK가 비즈니스 값 + 컬럼명이 `sn`일 때만 예외 (`@Id id IDENTITY` + `sn` 일반 컬럼). YAML `exclude-insert-columns: ["id"]` 로 기본값 오버라이드
