## 상세

### 오전

#### 추적 기능 점검 결과 (수정 전)

| 기능 | 결과 |
|------|------|
| `/tables` 매핑별 통계 | 정상 |
| `/data/source`, `/data/target-if` | 정상 |
| `/tables/{tableName}` 처리현황 클릭 | **실패** — 404 |
| trace-source 단일PK | 정상 |
| trace-source 복합PK | **실패** — SOURCE_NOT_FOUND |
| 정방향 추적 복합PK | **실패** — 부분매칭으로 여러건 반환 |

#### 버그 A: getTableLog 테이블명/매핑명 불일치
- Orchestrator가 테이블명(`if_snd_use_status_data`)을 보내는데 common은 매핑명(`use`)으로만 검색
- `findByExecutionIdAndMappingName` 실패 시 `sourceTables`/`targetTables` JSON에서 테이블명 fallback 검색 추가
- `getTableLog`, `getTableFailedInfo` 동일 적용

#### 버그 B: buildSourceFilter 복합PK 서브쿼리
- `SPLIT_PART(source_refs, ':', 4)` → 복합PK면 `20260325|SC001` 통째로 추출 → BIGINT 캐스팅 실패
- 복합PK 감지(`"|"` 포함) 시 서브쿼리 스킵 → 기존 배치 분할 경로로 전환
- 배치모드에서도 복합PK용 `(col1=? AND col2=?) OR (...)` WHERE 생성

#### 버그 C: SOURCE 정방향 추적 — source_refs 기반 통일
- 기존: pkValue(첫 컬럼만) → LIKE 패턴 → 복합PK 부분매칭 → 여러건 반환
- 변경: target-if 데이터 조회 → source_refs PK 파싱 → 클릭한 행과 정밀 매칭 → 1건 반환
- 프론트에서 target-if 행을 직접 결과로 구성 (추가 API 호출 불필요)

#### 프론트 행 식별 변경
- 기존: 첫 컬럼 값 기준 → 복합PK에서 obsr_de 중복 시 여러 행 동시 펼침
- 변경: 인덱스 기반 → 행별 유니크 보장
