# bojo-int 자체 DB: PG → Oracle 이전 계획

## 목적
Internal Agent(sync-agent-bojo-int)의 자체 DB를 PG(29001/dev) → Oracle(29004/XEPDB1)로 이전.
실서버 내부망은 Oracle/Tibero만 사용하므로 PG 의존 제거.

## 현재 상태

| 항목 | 현재 | 목표 |
|------|------|------|
| bojo-int spring.datasource | PG 29001/dev | Oracle 29004/XEPDB1 |
| JPA dialect | PostgreSQLDialect | Oracle12cDialect |
| sync_log 테이블 | PG dev DB | Oracle XEPDB1 |
| execution 테이블 | PG dev DB | Oracle XEPDB1 |
| IF/Target | 이미 Oracle 29004 | 유지 |

## 영향 범위

- **sync-agent-bojo-int** 모듈만 (다른 모듈 영향 없음)
- **sync-agent-common** 엔티티는 공용이므로 Oracle 호환성 확보 필요 (bojo-int에서만 Oracle로 연결)
- PG dev DB에 있는 bojo-int의 sync_log/execution은 삭제 가능

## PG → Oracle 호환성 점검

### 1. JPA 엔티티 — 컬럼 타입

| 엔티티 | 컬럼 | 현재 (PG) | Oracle 이슈 | 대응 |
|--------|------|----------|------------|------|
| SyncLog | id | `@GeneratedValue(IDENTITY)` | Oracle 12c+ IDENTITY 지원 | DDL에 `GENERATED BY DEFAULT AS IDENTITY` |
| SyncLog | sourceTables, targetTables, failedKeys, errorSummary | `columnDefinition = "TEXT"` | Oracle TEXT 없음 | `VARCHAR2(4000)` 또는 `CLOB` |
| Execution | id | `@GeneratedValue(IDENTITY)` | 동일 | DDL에 IDENTITY |
| Execution | errorMessage | `columnDefinition = "TEXT"` | 동일 | VARCHAR2(4000) 또는 CLOB |
| 공통 | LocalDateTime | PG timestamp | Oracle TIMESTAMP | 호환 (문제 없음) |
| 공통 | Long/Integer | PG bigint/int | Oracle NUMBER | 호환 (문제 없음) |

> **결정 필요**: TEXT → VARCHAR2(4000) vs CLOB
> - VARCHAR2(4000): 단순, JPA 호환 좋음, 4000자 제한
> - CLOB: 무제한, 하지만 JPA에서 @Lob 필요할 수 있음
> - sourceTables/targetTables는 JSON 배열이라 4000자 충분 → VARCHAR2(4000)
> - errorMessage/failedKeys는 길어질 수 있음 → CLOB 또는 VARCHAR2(4000)

### 2. JPA Repository — @Query

| Repository | 메서드 | 쿼리 | Oracle 이슈 |
|-----------|--------|------|------------|
| SyncLogRepository | findFailedByExecutionId | 일반 비교 | **호환** |
| SyncLogRepository | sumCountsByExecutionId | COALESCE + SUM | **호환** |
| SyncLogRepository | findByExecutionIdAndSourceTableContaining | `LOWER() LIKE LOWER(CONCAT('%',...,'%'))` | Hibernate가 Oracle용으로 `||` 변환 → **호환** |
| SyncLogRepository | findByExecutionIdAndTargetTableContaining | 동일 | **호환** |
| ExecutionRepository | 커스텀 @Query 없음 | Spring Data 자동 | **호환** |

### 3. ExecutionDataController — JdbcTemplate native SQL (핵심)

이 컨트롤러는 **외부 datasource에 접속해서 실행 데이터를 조회**하는 용도이므로, 자체 DB(sync_log/execution)와는 별개. 이미 Oracle 분기가 구현되어 있음.

하지만 **sync_log를 JPA로 조회한 후 그 결과로 외부 DB 쿼리를 만드는 흐름**이므로, sync_log 자체가 Oracle에서 정상 읽히는지만 확인하면 됨.

| 항목 | 현재 상태 | 이슈 |
|------|----------|------|
| `detectDbType()` | Oracle 감지 구현됨 | **호환** |
| `isOracle()` | 구현됨 | **호환** |
| `qi()` | Oracle 분기 있음 | **호환** |
| `castToText()` | Oracle → TO_CHAR() | **호환** |
| `pagingSql()` | Oracle → OFFSET FETCH | **호환** |
| `buildSearchClause()` — ILIKE | PG만 ILIKE, Oracle은? | **이슈 — 아래 상세** |

#### ILIKE 이슈 (ExecutionDataController:1790)
```java
String likeOp = isMysql(dbType) ? "LIKE" : "ILIKE";  // Oracle에서 ILIKE 미지원!
```
- 현재: MySQL이면 LIKE, 그 외(PG 가정)는 ILIKE
- Oracle에서 ILIKE 사용 시 ORA 에러
- **수정**: Oracle도 LIKE 사용 + `UPPER()` 감싸기

#### TEXT CAST 이슈 (ExecutionDataController:1789)
```java
String castType = isMysql(dbType) ? "CHAR" : "TEXT";  // Oracle에서 TEXT 없음!
```
- **수정**: Oracle이면 `VARCHAR2` 또는 `TO_CHAR()` 사용

### 4. GimsTargetRepository — Oracle 이미 구현

bojo-int의 Loader Repository. IF/Target은 이미 Oracle이므로 문제 없음:
- `ensureResultId()`: Oracle SELECT→INSERT→SELECT 패턴 구현됨
- `batchUpsertLink()`: Oracle MERGE INTO 구현됨
- `batchInsertObsvdata()`: 일반 INSERT (호환)

### 5. SourceToIfStep — 외부 datasource용

자체 DB가 아닌 외부 datasource(IF/Target)에 대한 쿼리. Oracle 분기 이미 구현됨:
- `buildOracleMergeSql()`: 구현됨
- `castToText()`: Oracle TO_CHAR() 구현됨
- `convertParamForTarget()`: Oracle Time→String 처리 (오늘 수정)

## 작업 항목

### Step 1: common 엔티티 PG/Oracle 양쪽 호환 — columnDefinition 통일

SyncLog, Execution의 `columnDefinition = "TEXT"` → `length = 4000` 으로 변경.
DB 종류에 따라 Hibernate가 자동 매핑: PG → VARCHAR(4000), Oracle → VARCHAR2(4000).

```java
// 변경 전
@Column(name = "source_tables", columnDefinition = "TEXT")

// 변경 후 (PG/Oracle 양쪽 호환)
@Column(name = "source_tables", length = 4000)
```

**common이 PG/Oracle 양쪽에서 동작해야 하므로** DB 특정 columnDefinition 사용 금지.
- bojo, others, proxy → PG: VARCHAR(4000) (기존 TEXT와 내부적으로 동일, 읽기/쓰기 호환)
- bojo-int → Oracle: VARCHAR2(4000)

**대상 컬럼** (전부 String 타입):
| 엔티티 | 컬럼 | 현재 | 변경 |
|--------|------|------|------|
| SyncLog | sourceTables | `columnDefinition = "TEXT"` | `length = 4000` |
| SyncLog | targetTables | `columnDefinition = "TEXT"` | `length = 4000` |
| SyncLog | failedKeys | `columnDefinition = "TEXT"` | `length = 4000` |
| SyncLog | errorSummary | `columnDefinition = "TEXT"` | `length = 4000` |
| Execution | errorMessage | `columnDefinition = "TEXT"` | `length = 4000` |

> PG에서 VARCHAR(4000)과 TEXT는 성능 차이 없음 (내부적으로 varlena). 기존 테이블이 TEXT여도 Hibernate 읽기/쓰기에 문제 없음.

### Step 2: ExecutionDataController — ILIKE/TEXT cast Oracle 대응

```java
// ILIKE → Oracle 분기 추가
String likeOp;
if (isMysql(dbType)) likeOp = "LIKE";
else if (isOracle(dbType)) likeOp = "LIKE";  // UPPER() 감싸기
else likeOp = "ILIKE";  // PG

// TEXT cast → Oracle 분기
String castExpr;
if (isMysql(dbType)) castExpr = "CAST(" + col + " AS CHAR)";
else if (isOracle(dbType)) castExpr = "TO_CHAR(" + col + ")";
else castExpr = col + "::text";
```

### Step 3: Oracle DDL — sync_log + execution

```sql
-- scripts/oracle-init-self.sql
CREATE TABLE SYNC_LOG (
    ID NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    EXECUTION_ID VARCHAR2(100) NOT NULL,
    STEP_ID VARCHAR2(50),
    MAPPING_NAME VARCHAR2(100),
    SOURCE_TABLES VARCHAR2(4000),
    TARGET_TABLES VARCHAR2(4000),
    READ_COUNT NUMBER,
    WRITE_COUNT NUMBER,
    FAILED_COUNT NUMBER,
    SKIP_COUNT NUMBER,
    FAILED_KEYS VARCHAR2(4000),
    ERROR_SUMMARY VARCHAR2(4000),
    SOURCE_PK_COLUMN VARCHAR2(100),
    CREATED_AT TIMESTAMP DEFAULT SYSTIMESTAMP
);

CREATE TABLE EXECUTION (
    ID NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    EXECUTION_ID VARCHAR2(100) NOT NULL UNIQUE,
    AGENT_ID VARCHAR2(100),
    SOURCE_DATASOURCE_ID VARCHAR2(50),
    TARGET_DATASOURCE_ID VARCHAR2(50),
    PARENT_EXECUTION_ID VARCHAR2(100),
    STATUS VARCHAR2(20),
    STARTED_AT TIMESTAMP,
    FINISHED_AT TIMESTAMP,
    DURATION_MS NUMBER,
    TOTAL_READ_COUNT NUMBER,
    TOTAL_WRITE_COUNT NUMBER,
    TOTAL_SKIP_COUNT NUMBER,
    ERROR_MESSAGE VARCHAR2(4000)
);
```

## 리스크

| 리스크 | 영향 | 대응 |
|--------|------|------|
| common 엔티티 수정 → DMZ Agent 영향 | PG TEXT→VARCHAR(4000) | PG는 내부적으로 동일, 기존 테이블과 호환됨 |
| VARCHAR2(4000) 초과 데이터 | INSERT 실패 | failedKeys/errorSummary는 현실적으로 4000자 이내, 초과 시 truncate 처리 검토 |
| Hibernate Oracle dialect 호환 | JPA 쿼리 실패 | JPQL은 dialect 자동 변환, native query만 주의 |
| 기존 PG sync_log 데이터 | bojo-int 이력 소실 | 테스트 데이터이므로 무시 |
| DMZ Agent 회귀 | bojo/others 정상 동작 확인 필요 | 엔티티 수정 후 DMZ Agent 빌드+기동 테스트 |
