# Rollback — 롤백 절차 / 시나리오

배포 후 심각한 문제 발생 시 **이전 상태로 되돌리는 절차**.

## 1. 롤백 판단 기준

즉시 롤백 (Phase A/B 실패):
- 생존 헬스체크 실패 (서비스 기동 불가)
- 파이프라인 전면 중단 (Loader / SND 모두 실패)
- 데이터 손상 / 유실 가능성 발견

분석 후 판단 (Phase C/D 실패):
- 부분 기능 장애 (한 Agent / 한 파이프만 문제)
- 성능 저하 (즉각 손상 없음)
- 단발성 에러 (재실행으로 복구 가능)

## 2. 롤백 대상별 절차

### 2.1 JAR / 애플리케이션
- [ ] 이전 버전 JAR 백업 위치 확인
- [ ] 서비스 정지
- [ ] 이전 JAR 배치
- [ ] 서비스 기동
- [ ] 헬스체크 + 스모크

> **전제**: 배포 직전에 **반드시 이전 JAR 보관** (`production-checklist.md § 1.1` 확장 필요)

### 2.2 DDL / 스키마
- [ ] **롤백 스크립트 사전 작성** 원칙 (신규 컬럼/테이블 추가 시 drop 스크립트 동시 준비)
- [ ] 롤백 DDL 실행
- [ ] 데이터 유실 여부 확인
- [ ] 애플리케이션 재기동 (validate 통과 확인)

> **위험**: 데이터 유실 수반 가능 → 실행 전 백업 재확인

### 2.3 설정 / 환경변수
- [ ] 이전 환경변수 / yml 복원
- [ ] 서비스 재기동 (무중단 리로드 가능한 경우 제외)

### 2.4 Orchestrator DataSource 등록
- [ ] 이전 DataSource 등록 값 복원 (UI or 직접 SQL)
- [ ] 관련 Agent 재접속 확인

### 2.5 데이터 / 이미 적재된 레코드
- [ ] 백업 복구 (PITR 또는 dump 복구)
- [ ] 복구 범위 결정 (전체 vs 특정 테이블 / 시점)

## 3. 롤백 타임라인

| 단계 | 목표 시간 |
|------|---------|
| 판단 | 5 분 |
| 실행 개시 | 10 분 |
| 기본 기능 복구 | 30 분 |
| 전체 정상화 | 1 시간 |

## 4. 의사결정 매트릭스

| 증상 | 롤백 우선순위 | 대안 |
|------|:-----------:|------|
| 전 서비스 기동 불가 | **즉시** | — |
| Orchestrator 만 장애 | **즉시** (모든 Agent 관리 불가) | — |
| 특정 Agent 만 장애 | 분석 후 | 해당 Agent 만 이전 JAR + 다른 Agent 유지 |
| Proxy 장애 | **즉시** | — |
| API Provider 만 장애 | 분석 후 | 외부 영향 평가 → 단독 롤백 |
| 데이터 적재 이상 (중복 / 손상) | 즉시 | 재실행 멱등성 확인 후 재실행으로 복구 가능한지 판단 |
| 추적 기능 실패 | 분석 후 | 단독 롤백 (파이프 동작엔 영향 없음) |
| 성능 저하 | **롤백 지양** | 원인 분석 / 다음 배포 반영 |
